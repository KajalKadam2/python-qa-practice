Day 20 — Page Object Model
As your test suite grows, tests become fragile. When a button changes its label or a form moves to a new URL, you update it in 10 different test files. Page Object Model (POM) solves this — each page of your app has one Python class. Locators and actions live in the class. Tests just call methods.

When something changes, you fix it in one place. Every test that uses that page automatically gets the fix.
The problem POM solves
Without POM — duplicated everywhere
# test_login.py page.get_by_label("Username").fill(u) page.get_by_label("Password").fill(p) page.get_by_role("button").click() # test_security.py — same 3 lines again page.get_by_label("Username").fill(u) page.get_by_label("Password").fill(p) page.get_by_role("button").click()
With POM — one method, used everywhere
# test_login.py login.login(username, password) # test_security.py — same method login.login(username, password) # Button label changes? Fix ONCE in LoginPage. # All 10 tests fixed automatically.
Project structure you will build today
qa-automation/ ├── pages/ │ ├── __init__.py ← makes pages a package │ ├── base_page.py ← shared methods all pages inherit │ └── login_page.py ← LoginPage class ├── conftest.py ← fixtures using page objects ├── test_login_pom.py ← tests using LoginPage └── test_secure_pom.py ← tests using SecurePage
POM is the #1 design pattern asked about in every SDET interview. "How do you structure your test suite?" The answer is always Page Object Model. Today you build a real one.
1 of 9
BasePage — the parent all page objects inherit from
BasePage holds methods that every page needs — navigate, wait, take screenshot, check URL. All other page classes inherit from it so they get these methods for free. You never repeat navigation or wait logic across page classes.
Step 1 — Create the pages/ folder and files
mkdir pages touch pages/__init__.py touch pages/base_page.py touch pages/login_page.py
Step 2 — Write pages/base_page.py
# pages/base_page.py from playwright.sync_api import Page, expect class BasePage: """Base class for all page objects. Holds common navigation and utility methods.""" BASE_URL = "https://the-internet.herokuapp.com" def __init__(self, page: Page): self.page = page def navigate(self, path: str = "/"): """Navigate to a path on the base URL.""" self.page.goto(f"{self.BASE_URL}{path}") def get_title(self) -> str: """Return the current page title.""" return self.page.title() def get_url(self) -> str: """Return the current page URL.""" return self.page.url def expect_url(self, expected: str): """Assert the current URL matches expected.""" expect(self.page).to_have_url(expected) def take_screenshot(self, name: str): """Take a screenshot and save to screenshots/.""" self.page.screenshot(path=f"screenshots/{name}.png", full_page=True) def wait_for_url(self, url: str): """Wait until the page URL matches.""" self.page.wait_for_url(url)
__init__ stores the page object as self.page. Every method in BasePage and all child classes uses self.page to interact with the browser. This is the standard Python class pattern — the constructor stores what the class needs.
2 of 9
LoginPage — the first real page object
LoginPage inherits from BasePage and adds login-specific methods. Every locator and every action on the login page lives here. Tests never reference locators directly — they call methods.
Write pages/login_page.py
# pages/login_page.py from playwright.sync_api import Page, expect from pages.base_page import BasePage class LoginPage(BasePage): """Page object for the login page.""" URL = "/login" # ── Locators ───────────────────────────────── # Defined as properties so they are always fresh @property def username_input(self): return self.page.get_by_label("Username") @property def password_input(self): return self.page.get_by_label("Password") @property def login_button(self): return self.page.get_by_role("button", name="Login") @property def flash_message(self): return self.page.locator("#flash") # ── Actions ────────────────────────────────── def open(self): """Navigate to the login page.""" self.navigate(self.URL) return self def fill_username(self, username: str): self.username_input.fill(username) return self def fill_password(self, password: str): self.password_input.fill(password) return self def click_login(self): self.login_button.click() return self def login(self, username: str, password: str): """Complete login action in one call.""" self.fill_username(username) self.fill_password(password) self.click_login() return self # ── Assertions ─────────────────────────────── def expect_success(self): """Assert login succeeded.""" expect(self.flash_message).to_contain_text("secure area") return self def expect_error(self, text: str): """Assert error message contains text.""" expect(self.flash_message).to_contain_text(text) return self def expect_url_is_login(self): """Assert still on login page — login failed.""" self.expect_url(f"{self.BASE_URL}{self.URL}") return self
Returning self from every method enables method chaining: login_page.open().login("user","pass").expect_success(). Each method returns the page object so the next call flows from the previous one. This makes tests read like prose.
3 of 9
SecurePage — the page after login
Every page in your app gets its own page object. SecurePage represents the page you land on after a successful login. It follows the same pattern as LoginPage — locators as properties, actions as methods, assertions as methods.
Create pages/secure_page.py
# pages/secure_page.py from playwright.sync_api import Page, expect from pages.base_page import BasePage class SecurePage(BasePage): """Page object for the secure area.""" URL = "/secure" @property def heading(self): return self.page.get_by_role("heading", name="Secure Area", exact=True) @property def logout_button(self): return self.page.get_by_role("link", name="Logout") @property def flash_message(self): return self.page.locator("#flash") def logout(self): """Click logout and return to login page.""" self.logout_button.click() return self def expect_secure_area_visible(self): expect(self.heading).to_be_visible() return self def expect_logout_visible(self): expect(self.logout_button).to_be_visible() return self def expect_url(self): """Assert we are on the secure page.""" super().expect_url(f"{self.BASE_URL}{self.URL}") return self
super().expect_url(...) calls the parent BasePage.expect_url() method. This is Python inheritance — SecurePage reuses BasePage's implementation without duplicating it. When BasePage improves, all child classes benefit automatically.
4 of 9
conftest.py — fixtures that return page objects
Update your conftest.py so fixtures return page object instances instead of raw page. Tests then use the page object's methods instead of Playwright calls directly. This is the bridge between fixtures and POM.
Update conftest.py — add these fixtures
# conftest.py — add these below your existing fixtures from pages.login_page import LoginPage from pages.secure_page import SecurePage @pytest.fixture def login_page_obj(page: Page) -> LoginPage: """Return a LoginPage object navigated to /login.""" lp = LoginPage(page) lp.open() yield lp @pytest.fixture def secure_page_obj(page: Page) -> SecurePage: """Login and return a SecurePage object.""" lp = LoginPage(page) lp.open().login("tomsmith", "SuperSecretPassword!") sp = SecurePage(page) yield sp
Name POM fixtures differently from your existing ones — login_page_obj vs login_page. This lets you keep both styles while you transition. In a real project you would replace the old fixtures over time.

Tests using Page Object Model — what they look like
This is the payoff. Tests become so clean they read like plain English. No locators, no raw Playwright calls — just method names that describe what the test does.
Create test_login_pom.py
# test_login_pom.py import pytest from pages.login_page import LoginPage # ── Basic tests ─────────────────────────────────── def test_valid_login(login_page_obj: LoginPage): (login_page_obj .login("tomsmith", "SuperSecretPassword!") .expect_success()) def test_invalid_username(login_page_obj: LoginPage): (login_page_obj .login("wronguser", "wrongpass") .expect_error("Your username is invalid") .expect_url_is_login()) def test_invalid_password(login_page_obj: LoginPage): (login_page_obj .login("tomsmith", "wrongpass") .expect_error("Your password is invalid")) def test_empty_fields(login_page_obj: LoginPage): (login_page_obj .login("", "") .expect_error("Your username is invalid")) # ── Parametrized with POM ───────────────────────── @pytest.mark.parametrize("username, password, error", [ ("wronguser", "pass", "Your username is invalid"), ("tomsmith", "wrong", "Your password is invalid"), ("", "", "Your username is invalid"), ], ids=["wrong_user", "wrong_pass", "empty"]) def test_invalid_logins(login_page_obj: LoginPage, username, password, error): (login_page_obj .login(username, password) .expect_error(error))

Method chaining — why every method returns self
Every method in your page objects returns self. This enables chaining — each method call flows into the next. The result is test code that reads like a sentence describing what the test does.
Without vs with chaining
Without chaining — verbose
login_page_obj.fill_username("tomsmith") login_page_obj.fill_password("pass") login_page_obj.click_login() login_page_obj.expect_success()
With chaining — reads like prose
(login_page_obj .fill_username("tomsmith") .fill_password("pass") .click_login() .expect_success())
The key line that makes chaining work
def fill_username(self, username: str): self.username_input.fill(username) return self # ← this is ALL that enables chaining def fill_password(self, password: str): self.password_input.fill(password) return self # ← same # Now you can chain: # login_page_obj.fill_username("x").fill_password("y").click_login() # Each .method() operates on the same login_page_obj
What happens step by step
# login_page_obj.login("tomsmith", "pass").expect_success() # # 1. login_page_obj.login(...) runs fill + click, returns self (=login_page_obj) # 2. .expect_success() is called ON that returned self # 3. It asserts the flash message, returns self again # # It is equivalent to: result = login_page_obj.login("tomsmith", "pass") # result IS login_page_obj result.expect_success() # same object
Parentheses around a chained expression are needed in Python when you split it across lines: (obj.method1().method2()). Without them, Python treats each line as a separate expression.
