Type this — all range() patterns you will use
# range(stop) — 0 to stop-1 for i in range(3): print(f"Retry attempt {i + 1}") # 1, 2, 3 print() # range(start, stop) — start to stop-1 for i in range(1, 6): print(f"Test case {i}") # 1, 2, 3, 4, 5 print() # range(start, stop, step) — every 2nd number for port in range(8080, 8090, 2): print(f"Checking port: {port}") # 8080, 8082, 8084... print() # reversed range — count down for i in range(3, 0, -1): print(f"Countdown: {i}") # 3, 2, 1
Output
Retry attempt 1 / Retry attempt 2 / Retry attempt 3
Test case 1 ... Test case 5
Checking port: 8080 / 8082 / 8084 / 8086 / 8088
Countdown: 3 / 2 / 1
range() never includes the stop value — range(1, 6) gives 1, 2, 3, 4, 5. Think of stop as "stop before this number". The step argument lets you skip — range(0, 10, 2) gives even numbers 0 through 8.

break and continue — controlling loop flow used constantly in QA
break exits the loop immediately — stop searching once found. continue skips to the next iteration — skip invalid items, process valid ones. Both appear in almost every real automation script you will write.
break — stop as soon as you find what you need
# Find first failing test — stop searching after finding it results = [ {"name": "test_login", "status": "PASS"}, {"name": "test_register", "status": "PASS"}, {"name": "test_checkout", "status": "FAIL"}, {"name": "test_payment", "status": "FAIL"}, {"name": "test_logout", "status": "PASS"}, ] first_failure = None for result in results: if result["status"] == "FAIL": first_failure = result break # stop — no need to check the rest if first_failure: print(f"First failure: {first_failure['name']}") else: print("All tests passed!") # Find a user by ID — stop once found users = [{"id":1,"name":"George"},{"id":2,"name":"Janet"},{"id":3,"name":"Emma"}] target = None for u in users: if u["id"] == 2: target = u break print(f"Found: {target['name'] if target else 'not found'}")
Output
First failure: test_checkout
Found: Janet
continue — skip invalid items, process valid ones
# Skip empty or invalid emails — process only valid ones emails = ["george@reqres.in", "", "notanemail", "janet@reqres.in", None] print("Processing valid emails:") for email in emails: if not email: print(f" Skipping empty/None value") continue # skip to next item if "@" not in email: print(f" Skipping invalid: {email}") continue # skip to next item print(f" ✓ Processing: {email}") # only valid emails reach here
Output
Processing valid emails:
✓ Processing: george@reqres.in
Skipping empty/None value
Skipping invalid: notanemail
✓ Processing: janet@reqres.in
Skipping empty/None value
continue with guard clauses is the cleanest loop pattern. Check failure cases first, continue past them, and the actual processing logic sits at the bottom — unindented, readable, only reached by valid items. Same philosophy as function guard clauses from Day 5.

for / else — Python's secret weapon almost nobody knows this
Python's for loop has an optional else clause. The else block runs only if the loop completed without hitting a break. If break fired, else is skipped. This is exactly what you need for "search and report not found" patterns in QA.
Type this — feel how elegant it is
# Without for/else — needs a flag variable users = [{"id":1,"name":"George"},{"id":2,"name":"Janet"}] found = False for u in users: if u["id"] == 99: print(f"Found: {u['name']}") found = True break if not found: print("User 99 not found") # clunky print() # With for/else — no flag needed for u in users: if u["id"] == 99: print(f"Found: {u['name']}") break else: print("User 99 not found") # runs because break never fired print() # QA example — check a required element exists in a list required_fields = ["id", "email", "first_name"] api_response = {"id": 1, "email": "a@b.com"} # missing first_name! for field in required_fields: if field not in api_response: print(f"FAIL: missing required field '{field}'") break else: print("PASS: all required fields present")
Output
User 99 not found
 
User 99 not found
 
FAIL: missing required field 'first_name'
Read for/else as: "loop through everything — else (if nothing triggered break) run this." It eliminates the need for a boolean flag variable in search loops. Whenever you find yourself writing found = False before a loop, consider for/else instead.

while loops — retry logic and polling critical for automation
A while loop runs as long as a condition is True. In QA automation, you use while for: retry logic (retry a flaky action up to N times), polling (wait until a condition becomes true), and processing until done. Playwright's built-in waits handle most timing — but retry logic still needs while.
Pattern 1 — retry with max attempts
# Retry a flaky action — stop on success or max attempts import random random.seed(42) # makes random predictable for practice max_retries = 5 attempt = 0 success = False while attempt < max_retries and not success: attempt += 1 print(f"Attempt {attempt}/{max_retries}...") # Simulate: succeeds ~50% of the time success = random.random() > 0.5 if success: print(f" ✓ Succeeded on attempt {attempt}") else: print(f" ✗ Failed — retrying...") if not success: print(f"FAIL: action failed after {max_retries} attempts")
Output (your output varies — random)
Attempt 1/5...
✗ Failed — retrying...
Attempt 2/5...
✓ Succeeded on attempt 2
Pattern 2 — polling until ready (with timeout)
# Poll until a condition is met — with a timeout safety net import time max_wait_seconds = 5 poll_interval = 1 # check every 1 second elapsed = 0 is_ready = False print("Waiting for element to appear...") while elapsed < max_wait_seconds and not is_ready: time.sleep(poll_interval) # wait before checking elapsed += poll_interval # Simulate: element appears after 3 seconds is_ready = elapsed >= 3 print(f" {elapsed}s — element ready: {is_ready}") if is_ready: print("✓ Element found — continuing test") else: print(f"✗ FAIL: element not found after {max_wait_seconds}s")
Output (takes ~3 seconds)
Waiting for element to appear...
1s — element ready: False
2s — element ready: False
3s — element ready: True
✓ Element found — continuing test
Never use while True: without a guaranteed exit — it runs forever and hangs your tests. Always pair a while loop with a maximum count or timeout. Playwright's expect() handles UI waits better than manual polling — use while for retry logic on actions, not element waits.

zip — pairing two lists together data-driven testing
zip() pairs items from two lists by position — first with first, second with second. You use it when you have inputs and expected outputs as separate lists and want to loop over them together. It is how data-driven tests connect test data to expected results.
Type this
# Inputs and expected outputs as paired lists inputs = ["tomsmith", "wronguser", "", "tomsmith"] passwords= ["SuperSecretPassword!", "wrongpass", "pass", "wrongpass"] expected = ["secure area", "invalid", "invalid", "invalid"] print("Login test cases:") for i, (user, pw, exp) in enumerate(zip(inputs, passwords, expected), 1): print(f" Test {i}: user={user!r:12} expects='{exp}'") print() # Zip two lists to compare expected vs actual expected_statuses = [200, 201, 404, 200] actual_statuses = [200, 201, 500, 200] # 3rd is wrong! endpoints = ["/users", "/users", "/users/99", "/users/1"] print("API status checks:") for endpoint, expected, actual in zip(endpoints, expected_statuses, actual_statuses): icon = "✓" if actual == expected else "✗" detail = "" if actual == expected else f" (expected {expected}, got {actual})" print(f" {icon} {endpoint}{detail}") print() # zip stops at the shortest list a = [1, 2, 3, 4, 5] b = ["a", "b", "c"] # shorter list print(list(zip(a, b))) # [(1,'a'), (2,'b'), (3,'c')]
Output
Login test cases:
Test 1: user='tomsmith' expects='secure area'
Test 2: user='wronguser' expects='invalid'
Test 3: user='' expects='invalid'
Test 4: user='tomsmith' expects='invalid'
 
API status checks:
✓ /users
✓ /users
✗ /users/99 (expected 404, got 500)
✓ /users/1
 
[(1, 'a'), (2, 'b'), (3, 'c')]
In Week 3, you will replace separate inputs and expected lists with a list of dicts — cleaner for complex data. But zip is perfect for simple paired data and appears constantly in professional test code.

Nested loops — every combination cross-browser testing
A loop inside a loop runs the inner loop fully for every iteration of the outer loop. In QA this means: test every page on every browser, test every user with every permission, test every endpoint with every method. You already used this in the Day 3 challenge — today you master it properly.
Type this — cross-browser × cross-page matrix
browsers = ["chromium", "firefox", "webkit"] pages = ["/login", "/dashboard", "/checkout"] base_url = "https://staging.myapp.com" total_runs = 0 results = [] for browser in browsers: for page in pages: total_runs += 1 url = f"{base_url}{page}" print(f" [{browser:10}] Testing {url}") results.append("PASS") print(f"\nTotal: {total_runs} runs ({len(browsers)} browsers × {len(pages)} pages)") print(f"All passed: {all(r == 'PASS' for r in results)}")

[chromium ] Testing https://staging.myapp.com/checkout

... (9 lines total)

All passed: True
New built-in — all() and any()
# all() — True only if every item is truthy statuses = [200, 200, 201, 200] print(all(s < 300 for s in statuses)) # True — all are success results = ["PASS", "PASS", "FAIL", "PASS"] print(all(r == "PASS" for r in results)) # False — one FAIL # any() — True if at least one item is truthy print(any(r == "FAIL" for r in results)) # True — at least one FAIL print(any(s >= 500 for s in statuses)) # False — no server errors
all() and any() take a generator expression — like a one-line loop that produces True/False values. all(r == "PASS" for r in results) reads: "are all results PASS?". These replace manual loops with flags in most check-all / check-any scenarios.

QA Example 1 — Retry engine while + break + counter
Flaky tests — tests that sometimes pass and sometimes fail due to timing — are the #1 pain in automation. A retry engine wraps actions in a while loop with exponential-style backoff. This is exactly what real test frameworks implement.
Create day6_ex1.py
import random random.seed(7) def retry(action_name, max_attempts=3, success_rate=0.6): """Retry an action up to max_attempts times.""" attempt = 0 while attempt < max_attempts: attempt += 1 print(f" [{action_name}] Attempt {attempt}/{max_attempts}") # Simulate action success/failure succeeded = random.random() < success_rate if succeeded: print(f" ✓ Succeeded on attempt {attempt}") return True if attempt < max_attempts: print(f" ✗ Failed — waiting before retry...") print(f" ✗ GAVE UP after {max_attempts} attempts") return False # Run several actions through the retry engine actions = [ ("click_submit", 3, 0.7), ("wait_for_modal", 5, 0.4), ("verify_toast", 2, 0.9), ] passed = [] failed = [] for name, attempts, rate in actions: print(f"\nRunning: {name}") ok = retry(name, attempts, rate) (passed if ok else failed).append(name) print(f"\n--- Retry Summary ---") print(f"Succeeded: {passed}") print(f"Failed: {failed}")
Line (passed if ok else failed).append(name) is a one-line conditional append — pick which list to append to based on the result. It is terse but readable once you know the pattern. Perfectly valid Python.

QA Example 2 — Test matrix runner nested + zip + all/any
The complete cross-browser, cross-environment, cross-page matrix. Simulates what a real test runner does — iterates all combinations, collects results, produces a matrix report. This is the most production-like code you have written so far.
Create day6_ex2.py
browsers = ["chromium", "firefox"] envs = ["staging", "prod"] pages = ["/login", "/dashboard"] # Results matrix — dict of dicts matrix = {} import random random.seed(1) for env in envs: matrix[env] = {} for browser in browsers: matrix[env][browser] = {} for page in pages: # Simulate: prod has more failures rate = 0.7 if env == "prod" else 0.9 result = "PASS" if random.random() < rate else "FAIL" matrix[env][browser][page] = result # Print matrix report print(f"{'='*50}") print(f" Test Matrix Report") print(f"{'='*50}") all_results = [] for env, browsers_data in matrix.items(): print(f"\n Environment: {env.upper()}") for browser, pages_data in browsers_data.items(): env_results = list(pages_data.values()) all_results.extend(env_results) status = "✓ all pass" if all(r == "PASS" for r in env_results) else "✗ has failures" print(f" {browser:10}: {status}") for page, result in pages_data.items(): icon = "✓" if result == "PASS" else "✗" print(f" {icon} {page} — {result}") print(f"\n{'─'*50}") total = len(all_results) passed = all_results.count("PASS") print(f" Total: {passed}/{total} passed") print(f" Any failures: {any(r == 'FAIL' for r in all_results)}")
This is a three-level nested loop — envs → browsers → pages — producing a matrix of results stored in a nested dict. The reporting loop mirrors the data structure exactly. This pattern scales to any number of dimensions and is how real cross-browser test reports are generated.
