Function 1 safe_get — loop + try/except per key
for key in keys: try: current = current[key] except (KeyError, TypeError): return default
Loop through keys one at a time, return default on first failure. The loop lets you navigate any depth — 2 keys or 10 keys, same code. TypeError is caught alongside KeyError because if any level returns something that is not subscriptable (like None), Python raises TypeError not KeyError. Both mean "cannot go deeper" — both should return the default.
safe_get(None, "data") # → None, no crash
Handles None input gracefully. If the entire response dict is None (API returned nothing), None["data"] raises TypeError — which is caught. This means safe_get never crashes regardless of what is passed in. This is the robustness that API test helpers need.
Function 2 load_users — exception wrapping + per-user validation
except FileNotFoundError: raise TestDataError(f"File not found: '{filepath}'")
Catch low-level, raise high-level. The caller does not need to know about FileNotFoundError — they only need to know the test data could not be loaded. TestDataError with a clear message is more useful than a raw OS error. This is exception wrapping — convert implementation details into domain-meaningful errors.
missing = [f for f in required if f not in user] if missing: raise TestDataError(f"User at index {i} missing fields: {missing}")
Validate every user, report all missing fields at once. The comprehension collects all missing fields for that user before raising — so you see ['password', 'role'] not just the first missing field. The index i tells you exactly which user in the file is broken.
Function 3 safe_assert — re-raise pattern
except AssertionError: if on_fail == "raise": raise # re-raises the same exception print(f" ✗ FAIL: {message}") return False
Bare raise re-raises the current exception. When you catch an exception and then call raise with no argument, Python re-raises the exact same exception that was caught — same type, same message, same traceback. This is cleaner than raise AssertionError(message) which would create a new exception and lose context.
Function 4 run_with_teardown — finally guarantees teardown
try: test_fn() passed = True except AssertionError as e: error = f"AssertionError: {e}" except Exception as e: error = f"{type(e).__name__}: {e}" finally: if teardown_fn is not None: teardown_fn()
Two except blocks — specific before general. AssertionError is caught first for test failures. Exception catches everything else (crashes). Python checks excepts in order — the first match wins. If you reversed them, Exception would catch everything and AssertionError would never be reached. finally runs after both — teardown always happens.
try: teardown_fn() except Exception as e: print(f" ⚠ Teardown error: {e}")
Teardown itself is wrapped in try/except. If teardown crashes, it should not hide the original test failure. By catching teardown errors separately, both the test result and the teardown error are reported independently. Without this, a teardown crash would propagate up and mask the actual test failure.
The 5 mistakes most people made
Mistake 1 — safe_get catches only KeyError, misses TypeError
# Wrong — crashes when a nested value is None or not a dict except KeyError: return default # safe_get({"a": None}, "a", "b") → TypeError: 'NoneType' not subscriptable # Correct — catches both except (KeyError, TypeError): return default
When you navigate nested dicts, any level might be None or a non-dict. None["key"] raises TypeError, not KeyError. Catching both means safe_get handles any input without crashing — which is the entire point of a "safe" getter.
Mistake 2 — load_users raises generic Exception not TestDataError
# Wrong — caller must catch generic Exception except FileNotFoundError: raise Exception("File not found") # Correct — caller catches specific TestDataError except FileNotFoundError: raise TestDataError(f"File not found: '{filepath}'")
Custom exceptions make caller code cleaner and more precise. except TestDataError only catches data loading problems. except Exception catches everything — including bugs in your code that you did not intend to catch. Custom exceptions are not optional in professional code.
Mistake 3 — safe_assert always raises, ignores on_fail parameter
# Wrong — on_fail="log" still crashes except AssertionError as e: raise # always raises regardless of on_fail # Correct — only raise if on_fail=="raise" except AssertionError: if on_fail == "raise": raise print(f" ✗ FAIL: {message}") return False
The on_fail parameter controls the behaviour. "log" means collect and continue — soft assertion. "raise" means stop immediately — hard assertion. Both are useful: soft assertions let a test check multiple things before failing; hard assertions stop at the first critical failure.
Mistake 4 — teardown outside finally, skipped on test crash
# Wrong — teardown only runs if no exception try: test_fn() except Exception as e: error = str(e) teardown_fn() # NOT in finally — skipped if except re-raises # Correct — teardown in finally, always runs try: test_fn() except Exception as e: error = str(e) finally: teardown_fn() # ALWAYS runs
This is the most critical mistake of the challenge. Code after a try/except block only runs if the except block did not re-raise. Code in finally runs unconditionally. Teardown must be in finally — otherwise a crash inside except leaves your test environment dirty.
Mistake 5 — not wrapping teardown in its own try/except
# Risky — teardown crash hides the test failure finally: teardown_fn() # if this crashes, original error is lost # Safe — teardown errors reported separately finally: try: teardown_fn() except Exception as e: print(f" ⚠ Teardown error: {e}")
A crash in teardown should not mask the original test failure. If teardown raises an exception inside finally, it replaces the original exception — you lose the test failure information. Wrapping teardown in its own try/except means both errors are reported independently.
