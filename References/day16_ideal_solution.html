The decisions that matter most
# yield page — not return page @pytest.fixture def login_page(page: Page): page.goto(f"{BASE_URL}/login") yield page # ← yield, not return
yield instead of return enables teardown. With return, the fixture has no teardown — once it returns, it is done. With yield, everything after yield runs when the test finishes — whether passed or failed. Even simple fixtures should use yield as a habit so teardown can be added later without restructuring.
# BASE_URL defined once at the top BASE_URL = "https://the-internet.herokuapp.com" page.goto(f"{BASE_URL}/login") page.goto(f"{BASE_URL}/checkboxes")
Single source of truth for the base URL. When you switch from staging to production, you change one line. If you hardcoded the full URL in every fixture, switching environments means updating 4+ places and risking inconsistency. This is the same principle as Day 4 dicts applied to test configuration.
# No page.goto() in any test file def test_valid_login(login_page: Page, valid_user): login_page.get_by_label("Username").fill(valid_user["username"]) # ↑ test starts here — no navigation code
Tests contain only assertions — no setup. This is the entire point of fixtures. A test should read as: "given I am on the login page, when I fill the form, then I see the success message." Navigation is setup, not the test itself. When tests contain goto(), they are mixing concerns.
# valid_user fixture returns a dict @pytest.fixture def valid_user(): return { "username": "tomsmith", "password": "SuperSecretPassword!", "success_msg": "You logged into a secure area!" }
Data fixtures use return, not yield. Only fixtures that need teardown use yield. Data fixtures — ones that just provide test data — use return. No cleanup needed for a dict. Using yield for data fixtures is not wrong but adds unnecessary noise.
@pytest.fixture(autouse=True) def screenshot_on_failure(page: Page, request): yield if hasattr(request.node, "rep_call") and request.node.rep_call.failed:
hasattr check before accessing rep_call. If a test is skipped or errors during setup (not during the test call itself), rep_call may not exist on request.node. The hasattr guard prevents an AttributeError in the teardown — without it, a setup failure would cause the screenshot fixture itself to crash, producing a confusing secondary error.
first = checkboxes_page.locator("input[type='checkbox']").nth(0)
.nth(0) to pick a specific element from multiple matches. The page has two checkboxes — locator("input[type='checkbox']") matches both. .nth(0) picks the first, .nth(1) picks the second. This is the correct way to handle multiple matching elements when you know the position you want.
The 5 mistakes most people made
Mistake 1 — using return instead of yield in page fixtures
# Wrong — no teardown possible @pytest.fixture def login_page(page: Page): page.goto(f"{BASE_URL}/login") return page # teardown never possible # Correct — teardown can be added any time @pytest.fixture def login_page(page: Page): page.goto(f"{BASE_URL}/login") yield page # teardown can go below this line
Always use yield in page fixtures. Even if you have no teardown today, using yield means you can add cleanup later without changing the fixture structure. return and yield produce the same test result right now — but yield is the professional habit.
Mistake 2 — page.goto() inside test functions
# Wrong — test contains navigation (setup) def test_valid_login(login_page: Page, valid_user): login_page.goto("https://the-internet.herokuapp.com/login") # redundant! login_page.get_by_label("Username").fill(valid_user["username"]) # Correct — fixture already navigated def test_valid_login(login_page: Page, valid_user): login_page.get_by_label("Username").fill(valid_user["username"])
The fixture already called goto(). Adding it again in the test is redundant — it just navigates to the same page twice. When you use a fixture, trust it to have done its setup. Your test starts from whatever state the fixture left the page in.
Mistake 3 — missing hasattr check in screenshot fixture
# Risky — crashes if test errors during setup if request.node.rep_call.failed: # AttributeError if rep_call missing # Safe — checks first if hasattr(request.node, "rep_call") and request.node.rep_call.failed:
rep_call only exists after the test call phase. If a fixture fails during setup, the test never runs, so rep_call is never set. Without hasattr, the screenshot fixture crashes with AttributeError — adding a confusing secondary failure on top of the real one.
Mistake 4 — importing conftest fixtures manually
# Wrong — conftest fixtures never need importing from conftest import login_page, valid_user # ERROR # Correct — just use them as parameters def test_valid_login(login_page: Page, valid_user): # pytest injects them automatically
conftest.py is not a regular module. pytest handles fixture injection automatically — importing from conftest actually breaks things. Just add the fixture name as a function parameter and pytest finds and injects it. No import, ever.
Mistake 5 — hardcoding full URLs in fixtures instead of using BASE_URL
# Fragile — 4 fixtures to update when URL changes @pytest.fixture def login_page(page: Page): page.goto("https://the-internet.herokuapp.com/login") @pytest.fixture def checkboxes_page(page: Page): page.goto("https://the-internet.herokuapp.com/checkboxes") # Correct — one change propagates everywhere BASE_URL = "https://the-internet.herokuapp.com" @pytest.fixture def login_page(page: Page): page.goto(f"{BASE_URL}/login")
BASE_URL is the single source of truth. When you switch to a staging or production URL, one change in conftest.py updates every fixture. With hardcoded URLs, you have 4+ places to update — and always miss one.
