Part 1 Storing full context in each result dict
run_results.append({ "browser": browser, "test_id": tc["id"], "test_name": tc["name"], "priority": tc["priority"], "retries": tc["retries"], "status": status })
Store everything you will need later — not just the answer. Including priority and retries means Parts 2, 4, and 5 can filter and analyse without needing to look back at test_cases. In real automation projects, result objects carry the full context of a run — browser, test metadata, duration, environment — so reports and retry logic have everything they need in one place.
Part 2 continue to skip, while to retry, mutate in place
for result in run_results: if result["status"] != "FAIL": continue # guard — skip passing tests while attempt < max_r and result["status"] == "FAIL": ... if retry_pass: result["status"] = "PASS" # mutate in place
Three things working together. continue skips passing results cleanly — no nesting. while retries the specific failing result up to its allowed count. result["status"] = "PASS" mutates the dict in place — because run_results holds references to the same dicts, the change is visible everywhere. This means Part 5's summary reflects post-retry statuses automatically.
while attempt < max_r and result["status"] == "FAIL":
Two exit conditions in the while. The loop stops when attempts are exhausted OR when the test passes — whichever comes first. Writing only while attempt < max_r would keep retrying even after a successful retry. Both guards are needed.
Part 3 continue as a guard — clean priority filter
for tc in test_cases: if tc["priority"] == "P3": print(f" Skipped : {tc['name']}") continue print(f" Would run: {tc['name']}")
Guard clause with continue — the cleanest filter pattern. Check the skip condition first, print and continue, leave the action at the bottom unindented. The alternative — if tc["priority"] != "P3": print("Would run...") — works but adds a level of indentation for every line inside. For a single action this is fine; for ten actions inside the loop, guards keep it flat and readable.
Part 4 for/else — the right tool for "find or report not found"
p1_results = [r for r in run_results if r["priority"] == "P1"] for result in p1_results: if result["status"] == "FAIL": print(f" First P1 failure: ...") break else: print(" All P1 tests passed!")
Pre-filtering before for/else makes the intent clear. Filtering to p1_results first means the loop only considers P1 tests — the for/else then purely answers "did any of these fail?". Mixing the priority filter inside the for/else loop would work but obscures the structure. Separate concerns: filter first, then search.
Part 5 all() and any() with conditions on filtered data
all_p1_passed = all( r["status"] == "PASS" for r in run_results if r["priority"] == "P1" ) any_failed = any(r["status"] == "FAIL" for r in run_results)
Generator expressions inside all() and any() can include a filter. all(condition for r in list if filter) — the if at the end limits which items are checked. This avoids a separate pre-filter step when you only need the boolean result. sum(1 for r in run_results if r["status"] == "PASS") in the count section uses the same pattern — sum counts True items by treating each match as 1.
The 5 mistakes most people made
Mistake 1 — while loop missing the status condition
# Wrong — keeps retrying even after success while attempt < max_r: attempt += 1 retry_pass = random.random() > 0.5 if retry_pass: result["status"] = "PASS" # Loop continues even though test already passed! # Correct — exit as soon as status is no longer FAIL while attempt < max_r and result["status"] == "FAIL": attempt += 1 ...
Every while loop needs both exit conditions. Without the status check, a recovered test would be retried again unnecessarily — potentially flipping back to FAIL. The and result["status"] == "FAIL" condition is the early exit that makes retry logic correct. Always ask: "what are all the reasons this loop should stop?"
Mistake 2 — not mutating result in place, creating a new variable instead
# Wrong — creates a local variable, doesn't update run_results if retry_pass: status = "PASS" # new local variable — run_results unchanged! # Correct — mutate the dict that's inside run_results if retry_pass: result["status"] = "PASS" # updates the dict in run_results
When you loop over a list of dicts, each iteration variable IS the dict — not a copy. Modifying result["status"] modifies the original dict inside run_results. Creating a new variable status = "PASS" only creates a local name — the dict in the list is untouched. Part 5's summary then reads stale statuses.
Mistake 3 — using if/else instead of for/else in Part 4
# Needs a flag — extra variable, extra logic found_failure = False for result in p1_results: if result["status"] == "FAIL": print(f"First P1 failure: ...") found_failure = True break if not found_failure: print("All P1 tests passed!") # for/else — no flag needed for result in p1_results: if result["status"] == "FAIL": print(f"First P1 failure: ...") break else: print("All P1 tests passed!")
for/else exists precisely to eliminate the flag variable pattern. Both versions produce identical output — but the flag version requires an extra variable, an extra assignment inside the loop, and an extra if-check after the loop. for/else expresses the same intent in fewer lines with no state to track.
Mistake 4 — not including retries field in result dict (Part 1)
# Missing — Part 2 can't access retries without going back to test_cases run_results.append({ "browser": browser, "test_id": tc["id"], "status": status # no priority, no retries — Part 2 and 4 will be harder }) # Complete — carry all needed context forward run_results.append({ "browser": browser, "test_id": tc["id"], "test_name": tc["name"], "priority": tc["priority"], "retries": tc["retries"], "status": status })
Design your data structures for what comes next, not just what you need right now. Part 1 creates the data. Parts 2–5 consume it. If you omit retries in Part 1, Part 2 must zip or look up back into test_cases — messy and fragile. Store everything downstream parts need when the data is created.
Mistake 5 — using a loop with a counter instead of sum() or all()/any()
# Verbose — manual counter loop total_passed = 0 for r in run_results: if r["status"] == "PASS": total_passed += 1 all_p1_passed = True for r in run_results: if r["priority"] == "P1" and r["status"] != "PASS": all_p1_passed = False break # Pythonic — one expression each total_passed = sum(1 for r in run_results if r["status"] == "PASS") all_p1_passed = all(r["status"] == "PASS" for r in run_results if r["priority"] == "P1")
Replace loop-with-counter with sum(), and loop-with-flag with all()/any(). Both versions work — but the Pythonic versions are shorter, have no mutable state to track, and read like prose. You will see these patterns in every professional Python codebase. Recognising them and writing them fluently is what separates Python beginners from intermediate developers.
