Every key decision explained
Part 1 List comprehension for URLs
full_urls = [f"{base_url}{page}" for page in test_pages]
Why comprehension, not a loop? When you are building a new list by transforming every item in an existing list — comprehension is the right tool. Read it as: "give me f"{base_url}{page}" for each page in test_pages." One line instead of four. The result is identical to appending inside a loop — just cleaner.
Part 2 Loop inside a loop — the critical pattern
for i, browser in enumerate(browsers, start=1): print(f"Browser {i}/{len(browsers)}: {browser.upper()}") for url in full_urls: print(f" [{browser}] Testing: {url}") results.append("PASS")
Outer loop = browsers. Inner loop = URLs. For every browser, we run through all 4 URLs. That gives 3 × 4 = 12 total runs. The results.append() is inside the inner loop — it fires once per test run. enumerate(browsers, start=1) counts from 1 so we print "Browser 1/3", not "Browser 0/3". browser.upper() on the header line makes the browser name stand out visually.
Part 3 Collecting results inside the loop
# results list created BEFORE the loop results = [] # append() called INSIDE the inner loop results.append("PASS") # count() called AFTER both loops finish n_passed = results.count("PASS") n_failed = results.count("FAIL")
Order matters. The empty list must exist before the loop starts. Append runs during the loop. Count runs after. Many people either create the list inside the loop (resets each iteration) or call count before the loop finishes (incomplete data). The pattern: create → fill → analyse — always in that order.
Part 4 Slicing and membership check
first_three = results[:3] # items 0, 1, 2 last_three = results[-3:] # last 3 items has_failure = "FAIL" in results
These three lines are pure list mechanics. [:3] gives first 3 — end index is excluded so it stops before index 3. [-3:] gives last 3 — negative index counts from end, empty end means "to the end of the list". "FAIL" in results returns a boolean — True if found anywhere, False if not. All three are O(n) operations Python handles internally — no loop needed.
Part 5 Dynamic status with inline condition
status = "ALL PASSED" if n_failed == 0 else f"{n_failed} FAILED" print(f"Browsers tested : {len(browsers)}") # len(), not hardcoded 3 print(f"Pages tested : {len(test_pages)}") # len(), not hardcoded 4 print(f"Total runs : {total_runs}") # computed, not hardcoded 12
Never hardcode values that can be computed. Writing print("Browsers tested : 3") breaks the moment you add a 4th browser to the list. len(browsers) is always correct regardless of list size. The inline condition for status goes further: if failures exist, it says how many — "2 FAILED" is more useful than just "FAILED".
The 5 mistakes most people made on this challenge
Mistake 1 — creating results list inside the loop
# Wrong — resets to empty every iteration for browser in browsers: results = [] # ← INSIDE loop, resets each time for url in full_urls: results.append("PASS") # Correct — create once, BEFORE the loop results = [] # ← OUTSIDE loop, persists for browser in browsers: for url in full_urls: results.append("PASS")
Why it matters: With the wrong version, results only ever holds 4 items (the last browser's runs) — never all 12. The list is silently overwritten each iteration. This is the most common loop bug and very hard to spot because no error is thrown.
Mistake 2 — building URLs with a loop instead of comprehension
# Works but verbose — 4 lines full_urls = [] for page in test_pages: full_urls.append(f"{base_url}{page}") # Preferred — 1 line, same result full_urls = [f"{base_url}{page}" for page in test_pages]
Both are correct. The loop version is not wrong — it's just an opportunity to practise comprehension. When you are transforming every item in a list into something else, comprehension is the Python-preferred pattern. You'll see it in every professional codebase.
Mistake 3 — hardcoding counts instead of computing them
# Fragile — breaks if list changes size print(f"Browsers tested : 3") print(f"Pages tested : 4") print(f"Total runs : 12") # Correct — always accurate print(f"Browsers tested : {len(browsers)}") print(f"Pages tested : {len(test_pages)}") print(f"Total runs : {len(results)}")
Hardcoded numbers are a maintenance trap. Add a 4th browser tomorrow and your summary still says "Browsers tested: 3". In real test suites this causes misleading reports. Always derive counts from the data itself — your code stays correct automatically.
Mistake 4 — wrong slicing for last 3
# Wrong — gets index 9,10,11 only if list has 12+ items last_three = results[9:12] # fragile — hardcoded positions # Wrong — gets all items from index 3 onward last_three = results[3:] # that's 9 items, not 3 # Correct — always the last 3 regardless of list size last_three = results[-3:] # negative index from the end
Negative slicing is the safe pattern. [-3:] means "start 3 from the end, go to the end." Works correctly for any list length. Hardcoded positions like [9:12] break if the list ever has fewer than 10 items.
Mistake 5 — static status string instead of dynamic
# Static — always says ALL PASSED even if tests fail print("Status : ALL PASSED") # Dynamic — reflects actual results status = "ALL PASSED" if n_failed == 0 else f"{n_failed} FAILED" print(f"Status : {status}")
A report that always says PASSED is useless. Drive every output from actual computed data. Try it: manually add "FAIL" to your results list and confirm the status changes. If it doesn't — your status is hardcoded.
