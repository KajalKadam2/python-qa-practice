Test 1 get_by_role — both Add and Delete use the same strategy
add_btn = page.get_by_role("button", name="Add Element") for _ in range(3): add_btn.click() expect(page.get_by_role("button", name="Delete")).to_have_count(3)
Store the locator in a variable when you use it multiple times. add_btn is evaluated once and reused in the loop — cleaner than calling get_by_role on every iteration. The Delete assertion uses to_have_count(3) which waits for exactly 3 Delete buttons — it handles dynamic page updates automatically without needing an explicit wait.
Test 2 get_by_label — assert URL not flash message
expect(page).to_have_url(f"{BASE}/secure")
URL assertion is more reliable than flash message for login success. The flash message can sometimes not appear if the page loads too fast or the element is dismissed. The URL change is definitive — if you are on /secure, login succeeded. This is the same lesson from Day 17 where to_have_url proved more reliable than to_have_title.
Test 3 Hovers link — get_by_role beats get_by_text
# Better — role makes the element type explicit page.get_by_role("link", name="Hovers", exact=True).click() # Also works but less specific page.get_by_text("Hovers", exact=True).click()

Test 4 CSS checkboxes — .first and .last over .nth()
checkboxes = page.locator("input[type='checkbox']") expect(checkboxes.first).not_to_be_checked() expect(checkboxes.last).to_be_checked()
Store the locator collection in a variable — avoid repeating the selector. checkboxes.first and checkboxes.last are more readable than checkboxes.nth(0) and checkboxes.nth(1) when you specifically mean the first and last items. Both work, but .first and .last communicate intent more clearly.
Test 5 XPath — use //* not specific element type
# Better — //* matches ANY element with that id page.locator("//*[@id='username']") # More fragile — breaks if element type changes from input to textarea page.locator("//input[@id='username']")
//* means "any element" — more resilient than specifying the tag. If a developer changes <input id="username"> to a custom component that renders differently, //*[@id='username'] still finds it. //input[@id='username'] would break. For ID-based XPath, always use //*.
Test 6 Dropdown — assert both value AND visible text
expect(page.locator("#dropdown")).to_have_value("2") expect(page.locator("#dropdown")).to_contain_text("Option 2")
Two assertions cover two different bugs. to_have_value("2") checks the HTML value submitted in forms. to_contain_text("Option 2") checks what the user sees on screen. A bug could swap the labels — value 2 shows "Option 1" — and a single assertion would miss it. You applied this correctly from Day 17 lesson.
The 5 mistakes most people made
Mistake 1 — using get_by_text for a link instead of get_by_role
# Fragile — matches any element containing "Hovers" page.get_by_text("Hovers").click() # Correct — explicitly matches only anchor elements page.get_by_role("link", name="Hovers", exact=True).click()
get_by_role is always preferred over get_by_text for clickable elements. The homepage has text in headings, paragraphs, and links. get_by_text("Hovers") could match a heading or paragraph too. get_by_role("link") restricts to anchor elements only — exactly what you want to click.
Mistake 2 — missing exact=True on get_by_role with name
# Can cause strict mode violations on pages with similar text page.get_by_role("link", name="Hovers").click() # Safe — matches only exactly "Hovers" page.get_by_role("link", name="Hovers", exact=True).click()
Without exact=True, name= does a contains match. A link named "Mouse Hovers" would also match name="Hovers". You hit this exact issue on Day 16 with the "Secure Area" heading. Make exact=True a habit for any name= argument when you want a specific element.
Mistake 3 — not storing the locator in a variable
# Repeating the same locator expression 3 times expect(page.locator("input[type='checkbox']").first).not_to_be_checked() page.locator("input[type='checkbox']").first.check() expect(page.locator("input[type='checkbox']").first).to_be_checked() # Correct — define once, reuse everywhere checkboxes = page.locator("input[type='checkbox']") expect(checkboxes.first).not_to_be_checked() checkboxes.first.check() expect(checkboxes.first).to_be_checked()
DRY applies to locators too. If the CSS selector ever changes, you update one variable not three lines. The code is also easier to read — checkboxes.first communicates intent better than the full repeated expression each time.
Mistake 4 — using //input in XPath instead of //*
# Fragile — breaks if element type changes page.locator("//input[@id='username']") # Resilient — works regardless of element type page.locator("//*[@id='username']")
//* matches any HTML element with that attribute. Modern frameworks often wrap inputs in custom components — the rendered HTML might be a div or span, not an input. Using //* future-proofs your XPath against element type changes while still targeting the exact element by ID.
Mistake 5 — asserting flash message instead of URL for login
# Flakier — flash message timing can vary expect(page.locator("#flash.success")).to_be_visible() # More reliable — URL change is definitive expect(page).to_have_url(f"{BASE}/secure")
URL is a more reliable success indicator than flash messages. Flash messages can be dismissed, fade out, or load at different speeds. The URL only changes when the server confirms login success and redirects. If you are on /secure, you are logged in — no ambiguity.
