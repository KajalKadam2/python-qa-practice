What is a dictionary — and why it is everywhere in QA
A dictionary stores data as key-value pairs — like a real dictionary where you look up a word (key) to find its definition (value). Instead of numbered positions like a list, you access values by a meaningful name.

In QA automation, dicts are used for: test user data (username, password, role), API JSON responses (every API returns a dict), test configuration (base URL, timeout, browser), expected vs actual results. When your Playwright test reads response.json()["data"]["email"] — it is reading a dict.
List vs Dict — when to use which
List — ordered items
browsers = ["chromium","firefox"] results = ["PASS","FAIL","PASS"] urls = ["/login","/dashboard"] Access by position: browsers[0] → "chromium" Use when: order matters, items are the same type
Dict — named fields
user = { "username": "tomsmith", "password": "Secret!", "role": "admin" } Access by name: user["username"] → "tomsmith" Use when: items have different meanings
Type this into day4.py
# Create a dict — curly braces, key: value pairs test_user = { "username": "tomsmith", "password": "SuperSecretPassword!", "role": "admin", "active": True, "login_count": 42 } # Access by key — use the name, not a number print(test_user["username"]) # tomsmith print(test_user["role"]) # admin print(test_user["active"]) # True print(test_user["login_count"]) # 42 # f-string with dict access print(f"Logging in as: {test_user['username']}") print(f"Role: {test_user['role']} | Active: {test_user['active']}") # Check the type print(type(test_user)) # <class 'dict'> print(len(test_user)) # 5 (number of key-value pairs)
Output
tomsmith
admin
True
42
Logging in as: tomsmith
Role: admin | Active: True
<class 'dict'>
5
Inside an f-string, use single quotes for dict keys when the f-string itself uses double quotes: f"User: {user['name']}". Mixing them wrongly causes a SyntaxError. The outer string uses double quotes, the inner key uses single quotes.

Safe access and dict methods
Accessing a key that doesn't exist crashes with a KeyError. The .get() method solves this — it returns a default value instead of crashing. In API testing, response fields are sometimes missing — .get() makes your tests resilient.
Type this
# Simulating an API response — some fields may be missing api_response = { "id": 2, "email": "janet.weaver@reqres.in", "first_name": "Janet" # "phone" is missing — not all APIs return all fields } # Direct access — crashes if key missing print(api_response["email"]) # OK # print(api_response["phone"]) # KeyError: 'phone' — CRASH # .get() — safe, returns None by default print(api_response.get("email")) # janet.weaver@reqres.in print(api_response.get("phone")) # None (no crash!) # .get() with a default value print(api_response.get("phone", "not provided")) # not provided print(api_response.get("id", 0)) # 2 (key exists, uses real value)
Output
janet.weaver@reqres.in
janet.weaver@reqres.in
None
not provided
2
The dict methods you will actually use
.get(k, default)
Safe access — returns default if key missing
d.get("phone","N/A") → "N/A"
.keys()
Returns all keys — check what fields exist
d.keys() → dict_keys(["id","email"])
.values()
Returns all values — check the data
d.values() → dict_values([2,"janet..."])
.items()
Returns key-value pairs — for looping
for k, v in d.items(): ...
"key" in d
Check if key exists — returns True/False
"email" in d → True
.update(other)
Merge another dict in — add/overwrite keys
d.update({"role":"admin"})
.pop(key)
Remove and return a key's value
d.pop("password") → "Secret!"
d[key] = val
Add or update a key
d["status"] = "verified"
The three most used in QA automation are .get() for safe API field access, .items() for looping over config, and "key" in dict for checking if a response field exists. Learn these three cold.

Modifying dicts + looping over them
Dicts are mutable — you can add, update, and remove keys after creation. Looping with .items() gives you both the key and value together, which is the standard way to print or process all fields in a test config or API response.
Type this — modifying a dict step by step
# Start with a basic config config = { "env": "staging", "browser": "chromium", "timeout": 5000 } print(f"Original: {config}") # Add a new key config["headless"] = True config["retries"] = 3 print(f"After add: {config}") # Update an existing key config["timeout"] = 8000 config["env"] = "prod" print(f"After update: {config}") # Remove a key removed = config.pop("retries") print(f"Removed 'retries' (value was {removed}): {config}") # Check if a key exists before using it if "headless" in config: print(f"Headless mode: {config['headless']}") # Loop over ALL key-value pairs print("\n--- Config dump ---") for key, value in config.items(): print(f" {key:10} : {value}")
Output
Original: {'env': 'staging', 'browser': 'chromium', 'timeout': 5000}
After add: {'env': 'staging', 'browser': 'chromium', 'timeout': 5000, 'headless': True, 'retries': 3}
After update: {'env': 'prod', 'browser': 'chromium', 'timeout': 8000, 'headless': True, 'retries': 3}
Removed 'retries' (value was 3): {'env': 'prod', ...}
Headless mode: True
 
--- Config dump ---
env : prod
browser : chromium
timeout : 8000
headless : True
{key:10} inside an f-string pads the key to 10 characters wide — lines up the colons perfectly in the output. Small detail, big readability improvement in test logs. The number after the colon is the column width.

Nested dicts — dicts inside dicts what real API responses look like
Real API responses are nested — a dict containing dicts containing lists. When reqres.in returns {"data": {"id": 2, "email": "..."}, "support": {"url": "..."}}, every level is a dict. You access nested fields by chaining square brackets.
Type this
# Simulating reqres.in/api/users/2 response response = { "data": { "id": 2, "email": "janet.weaver@reqres.in", "first_name": "Janet", "last_name": "Weaver", "avatar": "https://reqres.in/img/faces/2-image.jpg" }, "support": { "url": "https://contentcaddy.io?utm_source=reqres", "text": "Tired of writing endless social media content?" } } # Access nested fields — chain the brackets user_id = response["data"]["id"] email = response["data"]["email"] first = response["data"]["first_name"] last = response["data"]["last_name"] print(f"ID: {user_id}") print(f"Email: {email}") print(f"Name: {first} {last}") # Safe nested access with .get() phone = response["data"].get("phone", "not in response") print(f"Phone: {phone}") # Assert nested fields — what a real API test does assert response["data"]["id"] == 2, "User ID should be 2" assert "@" in response["data"]["email"], "Email should contain @" assert response["data"]["first_name"] == "Janet", "First name mismatch" print("All assertions passed!")
Output
ID: 2
Email: janet.weaver@reqres.in
Name: Janet Weaver
Phone: not in response
All assertions passed!
The pattern response["data"]["email"] is exactly what you write in Week 4 API tests after response.json(). The .json() method converts the API response into a Python dict — then you access it with these exact same brackets. You are writing real API assertion logic right now.

List of dicts — the most common API pattern appears in every real project
When an API returns multiple records — a list of users, a list of products, a list of orders — it sends a list of dicts. Each dict is one record. You loop over the list and access fields from each dict. This is the single most important data pattern in API test automation.
Type this
# Simulating reqres.in/api/users — list of user dicts users = [ {"id": 1, "email": "george.bluth@reqres.in", "first_name": "George", "active": True}, {"id": 2, "email": "janet.weaver@reqres.in", "first_name": "Janet", "active": True}, {"id": 3, "email": "emma.wong@reqres.in", "first_name": "Emma", "active": False}, {"id": 4, "email": "eve.holt@reqres.in", "first_name": "Eve", "active": True}, ] # Loop over list, access dict fields for each user print("All users:") for user in users: status = "ACTIVE" if user["active"] else "INACTIVE" print(f" [{status}] ID {user['id']}: {user['first_name']} — {user['email']}") # Filter — only active users active_users = [u for u in users if u["active"]] print(f"\nActive users: {len(active_users)}/{len(users)}") # Extract just the emails into a list all_emails = [u["email"] for u in users] print(f"Emails: {all_emails}") # Find a specific user by ID target_id = 3 found = None for user in users: if user["id"] == target_id: found = user break print(f"\nUser {target_id}: {found['first_name'] if found else 'not found'}")
Output
All users:
[ACTIVE] ID 1: George — george.bluth@reqres.in
[ACTIVE] ID 2: Janet — janet.weaver@reqres.in
[INACTIVE] ID 3: Emma — emma.wong@reqres.in
[ACTIVE] ID 4: Eve — eve.holt@reqres.in
 
Active users: 3/4
Emails: ['george.bluth@reqres.in', 'janet.weaver@reqres.in', ...]
 
User 3: Emma
The list comprehension [u["email"] for u in users] extracts one field from every dict into a flat list. You will use this constantly in API tests to get all IDs, all emails, all statuses — so you can assert on the whole collection at once.

QA Example 1 — Test configuration system real project pattern
Every real automation project has a central config dict. All tests import it and use values from it. Change one value in the config — every test updates automatically. No hardcoding URLs or timeouts scattered across 50 test files.
Create day4_ex1.py
# Central test configuration — one place to rule them all config = { "env": "staging", "base_url": "https://staging.myapp.com", "browser": "chromium", "timeout": 5000, "headless": True, "retries": 2, "screenshot_on_fail": True, "credentials": { "valid": {"username": "tomsmith", "password": "SuperSecretPassword!"}, "invalid": {"username": "wronguser", "password": "wrongpass"} } } # Build URLs from config — never hardcoded login_url = f"{config['base_url']}/login" dashboard_url = f"{config['base_url']}/dashboard" api_url = f"{config['base_url']}/api/v1" # Access nested credentials valid_user = config["credentials"]["valid"] invalid_user = config["credentials"]["invalid"] # Print the full config — loop over top-level keys print(f"{'='*35}") print(f" Test Configuration") print(f"{'='*35}") for key, value in config.items(): if key != "credentials": # skip nested dict for now print(f" {key:22}: {value}") print(f"{'─'*35}") print(f" Login URL : {login_url}") print(f" Valid user : {valid_user['username']}") print(f" Invalid user : {invalid_user['username']}")

QA Example 2 — API response field validator week 4 preview
This is what your Week 4 API tests will look like — take a dict (from a JSON response), check that required fields exist, validate their values, and assert. The only difference in Week 4 is the dict comes from response.json() instead of being hardcoded here.
Create day4_ex2.py
# Simulating: data = requests.get(url).json()["data"] api_data = { "id": 2, "email": "janet.weaver@reqres.in", "first_name": "Janet", "last_name": "Weaver", "avatar": "https://reqres.in/img/faces/2-image.jpg" } # Define what fields MUST exist in the response required_fields = ["id", "email", "first_name", "last_name", "avatar"] # Check all required fields are present missing = [f for f in required_fields if f not in api_data] if missing: print(f"FAIL: Missing fields: {missing}") else: print("PASS: All required fields present") # Validate field values checks = { "id is integer": isinstance(api_data["id"], int), "id is positive": api_data["id"] > 0, "email has @": "@" in api_data["email"], "first_name not empty": len(api_data["first_name"]) > 0, "avatar is https": api_data["avatar"].startswith("https"), "avatar is image": api_data["avatar"].endswith((".jpg",".png",".webp")), } print("\nField validation:") all_ok = True for check_name, result in checks.items(): icon = "✓" if result else "✗" print(f" {icon} {check_name}") if not result: all_ok = False print(f"\nOverall: {'PASS' if all_ok else 'FAIL'}")
Output
PASS: All required fields present
 
Field validation:
✓ id is integer
✓ id is positive
✓ email has @
✓ first_name not empty
✓ avatar is https
✓ avatar is image
 
Overall: PASS
The checks dict maps a human-readable description to a boolean expression. Looping over checks.items() gives you a clean, readable validation report. This pattern scales to 50 checks without adding complexity — just add more entries to the dict.
