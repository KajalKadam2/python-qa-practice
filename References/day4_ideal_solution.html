Part 1Field presence — the right way to check
missing = [f for f in required_fields if f not in user] status = "PASS" if not missing else f"FAIL — missing: {missing}"
List comprehension with not in. For each field name in required_fields, check if it is absent from the user dict. This builds a list of missing field names — empty if everything is present. if not missing is True when the list is empty (empty list = falsy). The status string then includes the actual missing field names — useful diagnostic output, not just "FAIL".
Part 2Collecting failures correctly
all_failures = [] # outer list — persists across all users for user in api_users: failures = [] # inner list — resets each user if "@" not in user["email"]: failures.append(...) if not user["first_name"].strip(): failures.append(...) all_failures.extend(failures) # add this user's failures to total
Two-list pattern: inner + outer. failures resets each iteration — it only holds that one user's issues. all_failures accumulates across all users and is used in Part 5 to determine overall status. .extend() adds all items from one list into another — unlike .append() which would add the whole list as a single nested item. This is the correct pattern for collecting issues across multiple records.
if not user["first_name"].strip():
Why .strip() before the empty check? A name containing only spaces — " " — would pass if not user["first_name"] as False (the string is not empty). But it is effectively empty for a name field. .strip() removes all whitespace first, so a spaces-only value becomes "" which is falsy. Always strip before checking emptiness on user-facing text fields.
Part 3Three comprehensions — each extracts something different
all_emails = [u["email"] for u in api_users] all_ids = [u["id"] for u in api_users] active_users = [u for u in api_users if u["active"]]
Notice the difference between the three. The first two extract a field from each dict — result is a flat list of values. The third keeps the whole dict when a condition is met — result is a filtered list of dicts. Then [u["first_name"] for u in active_users] extracts names from the filtered list. Composing comprehensions this way is standard Python — read it inside-out: filter first, then extract.
Part 4Safe access — why .get() not [ ]
phone = user.get("phone", "N/A")
This is the entire point of .get(). None of these users have a "phone" key. Using user["phone"] would crash with KeyError on the first user. .get("phone", "N/A") returns "N/A" silently when the key is absent. In real API tests, response schemas vary — some endpoints return optional fields, others omit them. .get() is your safety net for every optional field.
Part 5Summary — all counts computed, never hardcoded
active_count = len([u for u in api_users if u["active"]]) bad_emails = len([u for u in api_users if "@" not in u["email"]]) empty_names = len([u for u in api_users if not u["first_name"].strip()]) overall = "PASS" if not all_failures else f"FAIL ({len(all_failures)} issues)"
Pattern: len([comprehension]) counts filtered items. This is the cleanest way to count items matching a condition — filter with comprehension, wrap with len(). The overall status uses all_failures from Part 2 — not a new check — so it stays consistent with the detailed output above. If Part 2 found 3 issues, Part 5 reports 3 issues. They can never disagree.
The 5 mistakes most people made
Mistake 1 — using append instead of extend for collecting failures
# Wrong — appends a list INTO a list (nested) all_failures.append(failures) # all_failures = [["email invalid"], ["first_name empty"]] ← nested lists! # Correct — adds each item from failures into all_failures all_failures.extend(failures) # all_failures = ["email invalid", "first_name empty"] ← flat list
Why it matters: append(list) adds the whole list as a single item, creating a nested structure. extend(list) unpacks the list and adds each item individually, keeping it flat. len(all_failures) then correctly counts total issues rather than counting batches of issues.
Mistake 2 — placing failures = [] outside the loop
# Wrong — never resets, last user's failures bleed into next failures = [] for user in api_users: if "@" not in user["email"]: failures.append("email invalid") # User 1 passes — but failures still contains User 3's issues! # Correct — reset inside loop, before each user's checks for user in api_users: failures = [] # ← fresh start for each user if "@" not in user["email"]: failures.append("email invalid")
Opposite of the Day 3 mistake. In Day 3, results = [] needed to be OUTSIDE the loop. Here, failures = [] needs to be INSIDE the loop — one fresh list per user. The rule: if a variable accumulates across all iterations, create it before the loop. If it resets each iteration, create it inside.
Mistake 3 — checking empty string with == instead of not + strip
# Fragile — misses strings that are only spaces if user["first_name"] == "": failures.append("first_name is empty") # Also fragile — " " is not == "" but is effectively empty if len(user["first_name"]) == 0: failures.append("first_name is empty") # Correct — strip first, then truthiness check if not user["first_name"].strip(): failures.append("first_name is empty")
not string.strip() is the correct empty check for text fields. It handles empty string, whitespace-only strings, and newlines. == "" only catches true empty strings. In real API responses, names sometimes come back as " " (spaces) due to data entry errors — your test should catch those too.
Mistake 4 — re-running validation logic in Part 5 instead of reusing all_failures
# Redundant — runs the same checks twice # In Part 2 you already found 3 issues # In Part 5 you check again independently overall = "PASS" if bad_emails == 0 and empty_names == 0 else "FAIL" # Correct — drive overall from the failures already collected overall = "PASS" if not all_failures else f"FAIL ({len(all_failures)} issues)"
Single source of truth. If you add a new validation check in Part 2, the overall status in Part 5 automatically reflects it — because both use all_failures. If you duplicate the logic, you must update two places every time. In test automation, diverging checks create false confidence — one layer says PASS, another says FAIL.
Mistake 5 — using direct access [ ] for the phone field
# Crashes immediately — "phone" key does not exist phone = user["phone"] # KeyError: 'phone' # Correct phone = user.get("phone", "N/A")
The entire purpose of Part 4 is to practise this habit. Real API responses drop fields for users with incomplete profiles, optional features, or different account types. Any field that might be absent needs .get(). The crash happens silently on test run — you would only discover it when a user without a phone field enters your test data.
