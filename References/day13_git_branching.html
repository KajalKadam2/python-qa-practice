Day 13 — Git branching and pull requests
So far you have been pushing everything directly to main. That works for solo projects but every company uses branches. A branch is a separate copy of your code where you make changes safely — without affecting main until you are ready.

Today you learn the workflow every QA automation team uses: create a branch → write tests → open a PR → merge → done.
Why branches matter — the real scenario
Imagine you are adding 10 new Playwright tests for the checkout page. While you are working, a colleague pushes a fix for a broken login test directly to main.

Without branches — your half-finished checkout tests and their login fix collide. CI fails. Everyone is blocked.

With branches — you both work independently. When each piece is ready and tested, it gets merged. CI always stays green on main.

Every company you interview at uses this workflow. When they ask "how do you manage your test code changes?" — the answer is feature branches and pull requests. Today you learn to do it for real.

Creating and switching branches
A branch is just a pointer to a commit. Creating one is instant. The key commands are git branch to see branches, git checkout -b to create and switch, and git switch in newer Git versions.
Run these in your playwright-ui-tests folder
$ # See all branches — * shows current branch $ git branch $ # Create AND switch to a new branch in one command $ git checkout -b feature/add-dropdown-tests $ # Confirm you are on the new branch $ git branch
Output
* main ← before creating
 
Switched to a new branch 'feature/add-dropdown-tests'
 
* feature/add-dropdown-tests ← now here
main
Branch naming convention for QA projects
# Format: type/short-description feature/add-checkout-tests # new test files fix/login-test-wrong-locator # fixing a broken test refactor/extract-base-page # restructuring code chore/update-requirements # updating dependencies
Use lowercase with hyphens — no spaces, no underscores. The prefix (feature/, fix/) tells anyone reading the branch list exactly what kind of change it contains.

Working on a branch — the daily flow
Once you are on a branch, everything works exactly the same as before — edit files, git add, git commit, git push. The only difference is your commits go to the branch, not main.
Complete workflow — branch to push
# 1. Create branch $ git checkout -b feature/add-dropdown-tests # 2. Make your changes — edit test files in VS Code # (add a new test, fix a locator, whatever) # 3. Stage and commit — same as always $ git add . $ git commit -m "add dropdown selection tests with parametrize" # 4. Push branch to GitHub — first time needs -u $ git push -u origin feature/add-dropdown-tests # 5. After first push, just use git push $ git push
After git push -u
To https://github.com/KajalKadam2/playwright-ui-tests.git
* [new branch] feature/add-dropdown-tests -> feature/add-dropdown-tests
Switch between branches
# Switch to main $ git checkout main # Switch back to your feature branch $ git checkout feature/add-dropdown-tests # See all branches including remote ones $ git branch -a
Always check which branch you are on before committing — run git branch or look at your terminal prompt. Committing to the wrong branch is the most common Git mistake.

Pull Requests — proposing your changes
A Pull Request (PR) is a proposal to merge your branch into main. It shows exactly what changed, runs CI automatically, and gives teammates a chance to review before the code lands in main. Every professional team uses PRs — even solo developers use them for their own portfolio.
Create a PR on GitHub — step by step
1. Push your branch to GitHub
2. Go to your repo on GitHub — you will see a yellow banner: "feature/add-dropdown-tests had recent pushes"
3. Click Compare & pull request
4. Write a clear title: "Add parametrized dropdown selection tests"
5. Write a description — what you changed and why
6. Click Create pull request
7. GitHub Actions runs CI automatically on the PR
8. Once CI passes → click Merge pull request
9. Delete the branch — keep the repo clean
Good PR description template
## What changed - Added parametrized dropdown tests covering Option 1 and Option 2 - Used dropdown_page fixture from conftest.py - Both tests assert value AND visible label text ## Why Dropdown selection was not covered in the test suite ## Test output PASSED test_dropdown_selection[option_1] PASSED test_dropdown_selection[option_2]
Even for your personal portfolio repo, write real PR descriptions. Screenshots of your PRs with good descriptions go in your portfolio. A recruiter who sees a well-written PR knows you work like a professional.

After merging — keeping your local repo in sync
After a PR is merged on GitHub, your local main branch is behind. You need to pull the merged changes down and clean up the old branch.
Run these after merging on GitHub
# 1. Switch back to main $ git checkout main # 2. Pull the merged changes from GitHub $ git pull # 3. Delete the feature branch locally — it is merged, no longer needed $ git branch -d feature/add-dropdown-tests # 4. See clean branch list $ git branch
Output
Switched to branch 'main'
Already up to date. (or: Fast-forward, N files changed)
Deleted branch feature/add-dropdown-tests (was a1b2c3d).
* main ← only main remains
The complete daily workflow in one block
$ git checkout -b feature/my-new-tests # create branch # ... write tests ... $ git add . && git commit -m "add tests" # commit $ git push -u origin feature/my-new-tests # push branch # ... open PR on GitHub, CI passes, merge ... $ git checkout main # back to main $ git pull # sync merged changes $ git branch -d feature/my-new-tests # clean up

Merge conflicts — what they are and how to fix them asked in every interview
A merge conflict happens when two branches change the same line of the same file. Git cannot decide which version to keep so it marks both and asks you to choose. Conflicts sound scary — they are just a text editing task.
What a conflict looks like in a file
<<<<<<< HEAD BASE_URL = "https://staging.myapp.com" # your version on main ======= BASE_URL = "https://qa.myapp.com" # incoming version from branch >>>>>>> feature/update-base-url
How to read it
<<<<<<< HEAD — your current branch version (what you have)
======= — the divider between the two versions
>>>>>>> feature/... — the incoming branch version (what is coming in)
Fix it — three steps
# Step 1 — Open the conflicted file in VS Code # Delete the conflict markers and keep the version you want: # BEFORE (conflicted) <<<<<<< HEAD BASE_URL = "https://staging.myapp.com" ======= BASE_URL = "https://qa.myapp.com" >>>>>>> feature/update-base-url # AFTER (resolved — kept the qa version) BASE_URL = "https://qa.myapp.com" # Step 2 — Stage the resolved file $ git add conftest.py # Step 3 — Complete the merge $ git commit -m "resolve merge conflict in conftest.py"
VS Code shows conflicts with colour-coded buttons — "Accept Current Change", "Accept Incoming Change", "Accept Both Changes". Click the right one and the markers are removed automatically. You do not have to edit the markers manually.

The real QA team Git workflow
This is exactly how QA automation teams work at real companies. Each feature or fix gets its own branch, CI runs on every PR, and main is always deployable.
Real team scenario
# Monday: You start new test coverage for checkout page $ git checkout main && git pull # always start from latest main $ git checkout -b feature/checkout-page-tests # ... write tests across Monday and Tuesday ... $ git add . && git commit -m "WIP: checkout page tests" $ git push -u origin feature/checkout-page-tests # Wednesday: Colleague merges a change to conftest.py into main # You need to bring those changes into your branch $ git fetch origin # get latest from GitHub $ git merge origin/main # bring main changes into your branch # Thursday: Tests complete, open PR $ git add . && git commit -m "feat: checkout page full test coverage" $ git push # → Open PR on GitHub → CI runs → team reviews → merge # Friday: Clean up $ git checkout main && git pull $ git branch -d feature/checkout-page-tests
git fetch origin downloads changes from GitHub without applying them. git merge origin/main then applies them to your current branch. This keeps your branch up to date with what teammates are doing — preventing large conflicts at PR time.
