Part 1 Pass rate + suite status
pass_rate = round(passed / total * 100, 1) suite_status = "PASS" if pass_rate >= 80 else "FAIL"
Ternary for binary labels. Suite status is either PASS or FAIL — exactly two outcomes, simple condition. Ternary is the right tool. round(..., 1) keeps one decimal place — 75.0% not 75.00000000001% from floating point arithmetic. Always round percentages before displaying them.
Part 2 Response time analysis with min/max/sum
fastest = min(times) slowest = max(times) average = round(sum(times) / len(times), 1) slow_count = len([t for t in times if t > 2000])
Python built-ins min(), max(), sum() work directly on any list of numbers. No loop needed. sum(times) / len(times) is the mean — no library required. len([comprehension]) counts filtered items — the same pattern from Day 4. These four lines replace what would take 10+ lines of manual looping.
if average < 800: perf_label = "FAST" elif average < 1500: perf_label = "ACCEPTABLE" else: perf_label = "SLOW"
Three outcomes — use if/elif/else, not ternary. A chained ternary with three options works but becomes hard to read. When you have 3+ outcomes with different labels, if/elif/else is cleaner and easier to extend. The rule: ternary for 2 options, if/elif/else for 3 or more.
Part 3 Error classification with if/elif/else
error_lower = error.lower() if "timeout" in error_lower: label = "TIMEOUT" elif "500" in error_lower: label = "SERVER ERROR" else: label = "TEST FAILURE"
Three distinct categories — if/elif/else is the right choice. elif means once one branch matches, the others are skipped — an error containing both "timeout" and "500" gets labelled "TIMEOUT", not both. .lower() before the check makes it case-insensitive — "TIMEOUT" and "Timeout" and "timeout" all match. Always lowercase before checking for keywords in user-facing text.
Part 4 Guard-clause style environment check
warnings = [] if not session["base_url"].startswith("https"): warnings.append(...) if session["env"] == "prod": warnings.append(...) if session["browser"] not in supported_browsers: warnings.append(...) env_safe = not warnings
Three independent checks — three separate if statements, not if/elif. Unlike Part 3 where categories are mutually exclusive, here all three checks can fail simultaneously. Using elif would skip the second and third checks if the first fails. env_safe = not warnings is clean truthiness — an empty list is falsy, so not [] is True. A non-empty warnings list is truthy, so not [...] is False.
Part 5 Three-condition overall gate
overall = "PASS" if (pass_rate >= 80 and slow_count == 0 and env_safe) else "FAIL"
The parentheses make the compound condition readable. Three things must all be true: pass rate good enough, no slow tests, environment safe. and means all three must pass — one failure brings FAIL. This is how CI/CD quality gates work in real projects. Variables computed in earlier parts (pass_rate, slow_count, env_safe) are reused here — the single source of truth pattern from Day 4 applied to conditions.
The 5 mistakes most people made
Mistake 1 — using elif for independent environment checks
# Wrong — elif skips checks if first one fails if not session["base_url"].startswith("https"): warnings.append("URL not secure") elif session["env"] == "prod": # never checked if URL fails! warnings.append("Running on prod") # Correct — three separate if statements if not session["base_url"].startswith("https"): warnings.append("URL not secure") if session["env"] == "prod": warnings.append("Running on prod")
This is the most important condition lesson. Use elif when categories are mutually exclusive (one error message per error). Use separate if statements when checks are independent (all failures should be found and reported). Getting this wrong means silently missing environment problems.
Mistake 2 — not lowercasing before keyword checks in Part 3
# Fragile — misses "TIMEOUT", "TimeOut", etc. if "timeout" in error: label = "TIMEOUT" # Correct — lowercase first, then check error_lower = error.lower() if "timeout" in error_lower: label = "TIMEOUT"
Error messages come from many sources — browsers, servers, test frameworks — each with different capitalisation conventions. Lowercasing before checking is the same pattern you learned for flash messages in Day 2. Apply it consistently to any text you do not control.
Mistake 3 — computing average without round()
# Ugly floating point output average = sum(times) / len(times) print(f"Average: {average}ms") # Average: 1157.5000000000002ms ← ugly # Clean output average = round(sum(times) / len(times), 1) print(f"Average: {average}ms") # Average: 1157.5ms ← clean
Floating point arithmetic produces tiny errors. Division in Python sometimes returns values like 1157.5000000000002 due to how computers store decimal numbers. Always round() before displaying or comparing floats. The second argument to round is decimal places — round(x, 1) gives one decimal place.
Mistake 4 — duplicating the overall condition logic
# Redundant — recomputes what was already found overall = "PASS" if ( results.count("FAIL") == 0 and len([t for t in times if t > 2000]) == 0 ) else "FAIL" # Correct — reuse variables already computed overall = "PASS" if (pass_rate >= 80 and slow_count == 0 and env_safe) else "FAIL"
Single source of truth applies to conditions too. pass_rate, slow_count, and env_safe were already computed in Parts 1–4. Reuse them. Recomputing means two places to update if the threshold changes, and the risk that the Part 5 logic silently diverges from the detailed output above it.
Mistake 5 — using a chained ternary for 3+ performance labels
# Hard to read with 3 outcomes perf = "FAST" if average < 800 else "ACCEPTABLE" if average < 1500 else "SLOW" # Cleaner for 3+ outcomes if average < 800: perf_label = "FAST" elif average < 1500: perf_label = "ACCEPTABLE" else: perf_label = "SLOW"
The rule: ternary for 2 outcomes, if/elif/else for 3+. Chained ternaries work but force the reader to parse the whole line before understanding any of it. With if/elif/else, each branch is obvious at a glance. Code is read 10x more than it is written — optimise for reading, not writing.
