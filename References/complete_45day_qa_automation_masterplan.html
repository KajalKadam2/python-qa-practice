Day 18pytest fixtures + conftest.py
2.5h▶
Read: playwright.dev/python/docs/test-runners (pytest section)
A fixture is setup code that runs before your test — removes copy-paste from every test function. conftest.py is a special file where shared fixtures live — pytest finds it automatically.
Exercise — conftest.py with a navigate fixture
Create conftest.py in your tests folder:

import pytest
@pytest.fixture
def login_page(page):
   page.goto("https://the-internet.herokuapp.com/login")
   return page

Now in your test file — use def test_valid_login(login_page): instead of (page)
Remove all page.goto() calls from your tests — the fixture handles it

Notice: your tests are now 2 lines shorter each. That's the point.

Day 22API concepts — no coding today
2h▶
What is a REST API: a way for systems to talk to each other over HTTP.

HTTP Methods: GET (read), POST (create), PUT (update), DELETE (remove)
Status codes: 200 OK, 201 Created, 400 Bad Request, 401 Unauthorized, 404 Not Found, 500 Server Error
JSON: the data format — key-value pairs, same as Python dicts
Headers: metadata sent with the request (content-type, auth token)
Exercise — Postman exploration
Open Postman. Do these calls manually — no code:

• GET reqres.in/api/users?page=1 — read the JSON response. Find the list of users.
• GET reqres.in/api/users/2 — get one user. Note the structure.
• POST reqres.in/api/users — body: {"name": "Test User", "job": "QA Engineer"} — see the 201 response
• DELETE reqres.in/api/users/2 — see the 204 response

For each: write down the status code and what you'd assert in a test.

Exercise — Day 24: GET tests

import requests
BASE = "https://reqres.in/api"

def test_get_users_returns_200():
  r = requests.get(f"{BASE}/users?page=1")
  assert r.status_code == 200

def test_get_users_has_data():
  r = requests.get(f"{BASE}/users?page=1")
  data = r.json()
  assert len(data["data"]) > 0

def test_get_single_user():
  r = requests.get(f"{BASE}/users/2")
  assert r.json()["data"]["email"] == "janet.weaver@reqres.in"

32% of SDET interviews now include a live coding test. They are NOT LeetCode hard problems. They test: can you write a function, fix a bug in a test, write a class, do basic string/list operations. Exactly what you learned in weeks 1–2.

Practice these without looking anything up:
