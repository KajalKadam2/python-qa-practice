Day 17 — parametrize and data-driven tests
You wrote test_valid_login and test_invalid_login as two separate functions on Day 16. They do the same thing — fill a form, click Login, check a message — just with different data.

parametrize lets you write one test function and run it automatically with multiple data sets. One function, five scenarios, zero duplication. This is data-driven testing — the most powerful testing pattern you will use every day.
The problem it solves
Without parametrize — copy paste
def test_login_wrong_user(page): page.fill("wronguser") # 5 lines def test_login_wrong_pass(page): page.fill("tomsmith") # same 5 lines def test_login_empty(page): page.fill("") # same 5 lines again
With parametrize — one function
@pytest.mark.parametrize("user,pw,msg",[ ("wronguser", "pass", "invalid"), ("tomsmith", "wrong", "invalid"), ("", "", "invalid"), ]) def test_login(page, user, pw, msg): # 5 lines — runs 3 times
pytest runs the parametrized function once per row of data. 3 rows = 3 tests. 10 rows = 10 tests. Adding a new test case means adding one line of data — not copying an entire function.

Basic parametrize syntax — learn the pattern
The decorator goes directly above the function. The first argument is a string of parameter names separated by commas. The second argument is a list of tuples — one tuple per test run.
Create test_parametrize.py and type this
import pytest from playwright.sync_api import Page, expect # ── Single parameter ───────────────────────────── @pytest.mark.parametrize("path", [ "/login", "/checkboxes", "/dropdown", "/add_remove_elements", ]) def test_pages_load(page: Page, path): """Each path should load without error.""" page.goto(f"https://the-internet.herokuapp.com{path}") # page should not show an error — just verify title expect(page).to_have_title("The Internet") # ── Multiple parameters ────────────────────────── @pytest.mark.parametrize("username, password, expected", [ ("tomsmith", "SuperSecretPassword!", "secure area"), ("wronguser", "wrongpass", "Your username is invalid"), ("tomsmith", "wrongpass", "Your password is invalid"), ("", "", "Your username is invalid"), ]) def test_login_scenarios(page: Page, username, password, expected): page.goto("https://the-internet.herokuapp.com/login") page.get_by_label("Username").fill(username) page.get_by_label("Password").fill(password) page.get_by_role("button", name="Login").click() expect(page.locator("#flash")).to_contain_text(expected)
pytest test_parametrize.py -v --headed
Output — notice test IDs generated automatically

PASSED test_parametrize.py::test_pages_load[/checkboxes]
PASSED test_parametrize.py::test_pages_load[/dropdown]

The test ID in brackets is built from your parameter values — [tomsmith-SuperSecretPassword!-secure area]. When a test fails you immediately know which data set caused it without reading the code.
2 of 9
ids= — giving test cases readable names professional habit
Auto-generated test IDs like [tomsmith-SuperSecretPassword!-secure area] are long and hard to read in CI output. The ids= parameter lets you give each test case a short, descriptive name. This is the difference between output you can scan in 2 seconds and output you have to decode.
Type this
import pytest from playwright.sync_api import Page, expect @pytest.mark.parametrize("username, password, expected", [ ("tomsmith", "SuperSecretPassword!", "secure area"), ("wronguser", "wrongpass", "Your username is invalid"), ("tomsmith", "wrongpass", "Your password is invalid"), ("", "", "Your username is invalid"), ], ids=[ "valid_credentials", "wrong_username", "wrong_password", "empty_fields", ]) def test_login_scenarios(page: Page, username, password, expected): page.goto("https://the-internet.herokuapp.com/login") page.get_by_label("Username").fill(username) page.get_by_label("Password").fill(password) page.get_by_role("button", name="Login").click() expect(page.locator("#flash")).to_contain_text(expected)
Output — now readable at a glance
PASSED test_login_scenarios[valid_credentials]
PASSED test_login_scenarios[wrong_username]
PASSED test_login_scenarios[wrong_password]
PASSED test_login_scenarios[empty_fields]
Always use ids= in production test suites. When a CI/CD pipeline runs 200 tests and 3 fail, you need to find the failures in 2 seconds. [empty_fields] tells you immediately. [-SuperSecretPassword!-secure area0] tells you nothing.

parametrize + fixtures together the professional combo
Fixtures handle setup. Parametrize handles data. Together they give you clean, readable, data-driven tests where setup is automatic and data is explicit. This is the pattern every professional test suite uses.
Add this to test_parametrize.py
# test_parametrize.py — using login_page fixture from conftest.py import pytest from playwright.sync_api import Page, expect # login_page fixture from conftest.py handles navigation # parametrize handles the test data @pytest.mark.parametrize("username, password, expected_text", [ ("tomsmith", "SuperSecretPassword!", "secure area"), ("wronguser", "wrongpass", "Your username is invalid"), ("tomsmith", "wrongpass", "Your password is invalid"), ("", "", "Your username is invalid"), ("TOMSMITH", "SuperSecretPassword!", "Your username is invalid"), ], ids=[ "valid", "wrong_user", "wrong_pass", "empty", "uppercase_user" ]) def test_login(login_page: Page, username, password, expected_text): # login_page fixture already navigated to /login login_page.get_by_label("Username").fill(username) login_page.get_by_label("Password").fill(password) login_page.get_by_role("button", name="Login").click() expect(login_page.locator("#flash")).to_contain_text(expected_text)
pytest test_parametrize.py::test_login -v --headed
5 tests from 1 function

PASSED test_login[wrong_user]
PASSED test_login[wrong_pass]

The login_page fixture runs before each parametrized test — every scenario starts with a fresh navigation to /login. pytest creates a new browser context per test automatically.
4 of 9
pytest.param — marks and expected failures powerful pattern
pytest.param wraps a parameter set and lets you add metadata — a custom ID, or a mark like xfail (expected to fail) or skip. Use xfail for known bugs — the test is documented but does not block the suite.
Type this
import pytest from playwright.sync_api import Page, expect @pytest.mark.parametrize("username, password, expected", [ # Normal test cases ("tomsmith", "SuperSecretPassword!", "secure area"), ("wronguser", "wrongpass", "Your username is invalid"), # pytest.param with custom id pytest.param( "tomsmith", "wrongpass", "Your password is invalid", id="wrong_password_only" ), # pytest.param with xfail — known bug, documented pytest.param( " tomsmith", "SuperSecretPassword!", "secure area", id="leading_space_in_username", marks=pytest.mark.xfail(reason="Bug #42: spaces not trimmed") ), # pytest.param with skip pytest.param( "admin", "admin", "secure area", id="admin_user", marks=pytest.mark.skip(reason="admin user not set up in this env") ), ]) def test_login_with_marks(page: Page, username, password, expected): page.goto("https://the-internet.herokuapp.com/login") page.get_by_label("Username").fill(username) page.get_by_label("Password").fill(password) page.get_by_role("button", name="Login").click() expect(page.locator("#flash")).to_contain_text(expected)
Output
PASSED test_login_with_marks[tomsmith-...]
PASSED test_login_with_marks[wronguser-...]
PASSED test_login_with_marks[wrong_password_only]
XFAIL test_login_with_marks[leading_space_in_username]
SKIPPED test_login_with_marks[admin_user]
xfail means "this test is expected to fail — it documents a known bug". If it unexpectedly passes, pytest marks it XPASS (unexpected pass) — alerting you that the bug was fixed. This is far better than commenting out a failing test.

Storing parametrize data in conftest.py clean project structure
For large test suites, test data lives in conftest.py alongside fixtures — keeping test files clean. The data is defined once and reused across multiple test files.
Add to conftest.py
# conftest.py — add these constants at the top INVALID_LOGIN_CASES = [ pytest.param("wronguser", "wrongpass", "Your username is invalid", id="wrong_user"), pytest.param("tomsmith", "wrongpass", "Your password is invalid", id="wrong_pass"), pytest.param("", "", "Your username is invalid", id="empty"), ] DROPDOWN_OPTIONS = [ pytest.param("1", "Option 1", id="option1"), pytest.param("2", "Option 2", id="option2"), ]
Then use in test files
# test_login.py from conftest import INVALID_LOGIN_CASES from playwright.sync_api import Page, expect import pytest @pytest.mark.parametrize("username, password, expected", INVALID_LOGIN_CASES) def test_invalid_logins(login_page: Page, username, password, expected): login_page.get_by_label("Username").fill(username) login_page.get_by_label("Password").fill(password) login_page.get_by_role("button", name="Login").click() expect(login_page.locator("#flash")).to_contain_text(expected) # test_dropdowns.py from conftest import DROPDOWN_OPTIONS @pytest.mark.parametrize("value, label", DROPDOWN_OPTIONS) def test_dropdown_selection(dropdown_page: Page, value, label): dropdown_page.locator("#dropdown").select_option(value) expect(dropdown_page.locator("#dropdown")).to_have_value(value)
Unlike fixtures, data constants DO need an explicit import: from conftest import INVALID_LOGIN_CASES. Only fixture functions are injected automatically — plain Python variables must be imported normally.

A complete login test suite using parametrize with fixtures. This is exactly what Portfolio Project #1 looks like — the kind of file recruiters see and immediately understand.
Create test_login_parametrized.py
# test_login_parametrized.py import pytest from playwright.sync_api import Page, expect # ── Valid login scenarios ──────────────────────── @pytest.mark.parametrize("username, password", [ ("tomsmith", "SuperSecretPassword!"), ], ids=["valid_user"]) def test_valid_logins(login_page: Page, username, password): login_page.get_by_label("Username").fill(username) login_page.get_by_label("Password").fill(password) login_page.get_by_role("button", name="Login").click() expect(login_page.locator("#flash.success")).to_be_visible() expect(login_page).to_have_url("https://the-internet.herokuapp.com/secure") # ── Invalid login scenarios ────────────────────── @pytest.mark.parametrize("username, password, error_contains", [ ("wronguser", "SuperSecretPassword!", "Your username is invalid"), ("tomsmith", "wrongpass", "Your password is invalid"), ("", "SuperSecretPassword!", "Your username is invalid"), ("tomsmith", "", "Your password is invalid"), ("", "", "Your username is invalid"), ("TomSmith", "SuperSecretPassword!", "Your username is invalid"), ("tomsmith ", "SuperSecretPassword!", "Your username is invalid"), ("' OR '1'='1","anything", "Your username is invalid"), ], ids=[ "wrong_username", "wrong_password", "empty_username", "empty_password", "both_empty", "case_sensitive", "trailing_space", "sql_injection", ]) def test_invalid_logins(login_page: Page, username, password, error_contains): login_page.get_by_label("Username").fill(username) login_page.get_by_label("Password").fill(password) login_page.get_by_role("button", name="Login").click() expect(login_page.locator("#flash.error")).to_contain_text(error_contains) expect(login_page).to_have_url("https://the-internet.herokuapp.com/login")

QA Example 2 — Dropdown + checkbox parametrized reusable patterns
The same parametrize pattern works for any repeating test action — dropdown selection, checkbox states, form field validation. Once you see the pattern, you apply it everywhere.
Create test_elements_parametrized.py
# test_elements_parametrized.py import pytest from playwright.sync_api import Page, expect # ── Dropdown ───────────────────────────────────── @pytest.mark.parametrize("option_value, expected_value", [ ("1", "1"), ("2", "2"), ], ids=["select_option_1", "select_option_2"]) def test_dropdown_options(dropdown_page: Page, option_value, expected_value): dropdown_page.locator("#dropdown").select_option(option_value) expect(dropdown_page.locator("#dropdown")).to_have_value(expected_value) # ── Checkboxes ─────────────────────────────────── @pytest.mark.parametrize("index, initial_state, action", [ (0, False, "check"), # first box starts unchecked, check it (1, True, "uncheck"), # second box starts checked, uncheck it ], ids=["check_first", "uncheck_second"]) def test_checkbox_interaction(checkboxes_page: Page, index, initial_state, action): checkbox = checkboxes_page.locator("input[type='checkbox']").nth(index) # Assert initial state if initial_state: expect(checkbox).to_be_checked() else: expect(checkbox).not_to_be_checked() # Perform action if action == "check": checkbox.check() expect(checkbox).to_be_checked() else: checkbox.uncheck() expect(checkbox).not_to_be_checked()

Day 17 challenge — write it all from scratch
New file: test_day17_challenge.py. Use your existing conftest.py fixtures.
Build a complete parametrized test file
Write these 4 parametrized test functions:

Test 1 — test_pages_load
Parametrize 5 paths: /login, /checkboxes, /dropdown, /add_remove_elements, /hovers. Each should load and show title "The Internet". Use ids= with short names.

Test 2 — test_login_scenarios
Combine valid + invalid in one parametrized function using login_page fixture. At least 5 scenarios. Use ids=. Add one xfail case for a known edge case.

Test 3 — test_add_remove_elements
Go to /add_remove_elements. Parametrize: click Add Element 1 time, 2 times, 3 times. After each: assert the correct number of Delete buttons are visible using to_have_count(n).

Test 4 — test_dropdown_then_verify
Parametrize selecting Option 1 and Option 2. After selecting, assert both to_have_value() AND that the selected option text is visible on the page.
Test 3 is the most interesting — it uses parametrize to control how many times an action repeats and then asserts based on that count. This is data-driven testing at its most powerful — the data drives not just the input but the expected outcome.
