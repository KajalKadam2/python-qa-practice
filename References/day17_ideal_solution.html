Test 1 pages_load — URL assertion over title assertion
expect(page).to_have_url(f"https://the-internet.herokuapp.com{path}")
URL is more reliable than title for "did the page load" checks. Page titles vary across this site — some pages have "The Internet", others have empty titles, others have something different. The URL is always predictable — if the page loaded without redirect or error, the URL matches exactly what you navigated to. You discovered this yourself during debugging. Always assert the URL for page load verification.
Test 2 login_scenarios — xfail for known edge case
pytest.param( " tomsmith", "SuperSecretPassword!", "secure area", id="leading_space", marks=pytest.mark.xfail(reason="Bug: leading spaces not trimmed") )
xfail documents a known bug without blocking the suite. You expect this test to fail because the site does not trim leading spaces from usernames. Marking it xfail means: "we know about this, it is documented, do not fail the build over it." If the site ever fixes the bug, the test flips to XPASS — alerting you to update the mark. This is far better than a comment or a skipped test.
# ids= list must match parameter list length exactly ], ids=[ "valid", "wrong_user", "wrong_pass", "empty", "valid_again", "leading_space" ])
When mixing regular tuples and pytest.param, ids= must cover all items including pytest.param entries. pytest.param already has id= set, but if you use an ids= list on the decorator, it overrides all individual ids. Be consistent — either use ids= list for all, or use id= inside each pytest.param. Mixing them causes confusion.
Test 3 add_remove_elements — to_have_count
for _ in range(click_count): add_btn.click() expect(page.get_by_role("button", name="Delete")).to_have_count(click_count)
The parametrize data drives both the action AND the expected outcome. click_count controls how many times the button is clicked AND what count to assert. This is the most powerful parametrize pattern — the data does not just feed inputs, it feeds the expected result. to_have_count(n) asserts that exactly n elements match the locator — perfect for dynamic lists.
for _ in range(click_count): # _ means "I don't need this variable"
Using _ as the loop variable is a Python convention. When you need a loop but do not use the iteration variable, name it _. It signals to any reader: "this variable is intentionally unused — I am just looping N times." Using i when you never reference i inside the loop is misleading.
Test 4 dropdown_then_verify — value AND text assertion
expect(dropdown_page.locator("#dropdown")).to_have_value(value) expect(dropdown_page.locator("#dropdown")).to_contain_text(label)
Two assertions per test — value and visible text. to_have_value("1") checks the HTML value attribute of the selected option — what gets submitted in a form. to_contain_text("Option 1") checks what the user actually sees on screen. A dropdown could have the right value but display the wrong label — checking both catches that bug. This is your QA instinct in code.

Mistake 1 — asserting title instead of URL for page load
# Fragile — title varies across pages on this site expect(page).to_have_title("The Internet") # /add_remove_elements has empty title → test fails # Robust — URL always matches expect(page).to_have_url(f"https://the-internet.herokuapp.com{path}")
You discovered this yourself during the lesson. The page title is content controlled by the app — it can change, be empty, or vary per page. The URL is structural — if you navigated to it and it loaded, the URL matches. For "does the page load" checks, always assert the URL.
Mistake 2 — no xfail on the leading space edge case
# Wrong — test fails and blocks the suite (" tomsmith", "SuperSecretPassword!", "secure area"), # Or worse — test is commented out # (" tomsmith", "SuperSecretPassword!", "secure area"), # Correct — documented as known bug pytest.param( " tomsmith", "SuperSecretPassword!", "secure area", id="leading_space", marks=pytest.mark.xfail(reason="Bug: leading spaces not trimmed") )
Commented-out tests are invisible bugs. xfail tests are visible in the output, documented with a reason, and automatically flip to XPASS when the bug is fixed. A commented-out test is forgotten forever. Your 7.4 years of QA experience should make this instinctive — never suppress a known bug, document it.
Mistake 3 — clicking Add Element in a loop with index variable
# Unnecessary — index i is never used inside the loop for i in range(click_count): add_btn.click() # Correct Python convention for _ in range(click_count): add_btn.click()
_ signals "intentionally unused variable". When the loop variable is never referenced inside the body, naming it _ communicates intent clearly. Any reader immediately knows the loop is just for repetition — not for using the index value. Minor but marks professional Python awareness.
Mistake 4 — only asserting value, not label text in dropdown
# Incomplete — only checks the HTML value attribute expect(dropdown_page.locator("#dropdown")).to_have_value(value) # Complete — also checks what the user sees expect(dropdown_page.locator("#dropdown")).to_have_value(value) expect(dropdown_page.locator("#dropdown")).to_contain_text(label)
Value and label are different things. A developer could accidentally swap option labels — Option 1 appears when Option 2 is selected — and a value-only assertion would miss it. Both assertions together catch both bugs. You discovered this mismatch yourself during the lesson when the assertion failed.
Mistake 5 — missing ids= so test names are unreadable
# Without ids= — hard to read in CI output @pytest.mark.parametrize("path", ["/login", "/checkboxes", "/dropdown"]) # Output: test_pages_load[/login] test_pages_load[/checkboxes] # (OK here but breaks with complex data) # With ids= — always clear @pytest.mark.parametrize("path", [...], ids=["login", "checkboxes", "dropdown"]) # Output: test_pages_load[login] test_pages_load[checkboxes]
ids= is not optional in professional suites. With single simple strings it barely matters. With tuples of 3+ values the auto-generated ID becomes a wall of text. Always use ids= — it costs 30 seconds and saves minutes of debugging when a test fails in CI.
