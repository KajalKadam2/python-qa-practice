Day 20 — Page Object Model
You now have 20+ tests across 3 files. Every test directly calls Playwright methods — page.get_by_label("Username").fill(...). If the login form changes, you update every test that touches it.

Page Object Model fixes this. You create one class per page — LoginPage, CheckboxesPage — that wraps all the Playwright calls. Tests only call methods on those classes. When a locator changes, you update one class, not 20 tests.
The problem it solves
Without POM — locator in every test
# 20 tests each have this page.get_by_label("Username").fill(u) page.get_by_label("Password").fill(p) page.get_by_role("button", name="Login").click() # Form changes → update 20 tests
With POM — one class, clean tests
# 20 tests each have this login.login(username, password) # Form changes → update LoginPage # Zero test changes needed
Project structure you will build today
qa-automation/ ├── pages/ ← new folder │ ├── __init__.py ← makes pages a package │ ├── base_page.py ← shared methods all pages use │ ├── login_page.py ← login page actions + locators │ └── checkboxes_page.py ← checkboxes page ├── conftest.py ← fixtures updated to use POM ├── test_login_pom.py ← clean tests using POM └── test_checkboxes_pom.py
Create the pages/ folder inside your qa-automation folder right now before moving to the next page. You will create each file step by step.

BasePage — shared methods every page needs
BasePage is the parent class all page objects inherit from. It holds methods used across every page — navigate, wait for load, take screenshot, get title. Instead of duplicating these in every page class, they live in one place.
Create pages/base_page.py
# pages/base_page.py from playwright.sync_api import Page, expect class BasePage: """Base class for all page objects. Contains shared methods every page uses.""" def __init__(self, page: Page): self.page = page def navigate(self, url: str) -> "BasePage": """Navigate to a URL. Returns self for chaining.""" self.page.goto(url) return self def get_title(self) -> str: """Return the current page title.""" return self.page.title() def get_url(self) -> str: """Return the current page URL.""" return self.page.url def expect_url(self, url: str) -> "BasePage": """Assert the current URL matches.""" expect(self.page).to_have_url(url) return self def expect_title(self, title: str) -> "BasePage": """Assert the page title matches.""" expect(self.page).to_have_title(title) return self def screenshot(self, name: str) -> "BasePage": """Take a screenshot. Returns self for chaining.""" self.page.screenshot(path=f"screenshots/{name}.png", full_page=True) return self
Method chaining — returning self from every method lets you chain calls: page.navigate(url).expect_url(url).screenshot("loaded"). This is optional but makes test code read like a sentence. Standard pattern in professional page objects.
Create an empty pages/__init__.py file too — just create it with no content. This makes pages a Python package so you can import from it.

LoginPage — all login actions and locators in one place
LoginPage inherits from BasePage and adds everything specific to the login page — locators, actions, and assertions. Tests never touch a locator directly — they call login_page.login(username, password) and the page object handles everything.
Create pages/login_page.py
# pages/login_page.py from playwright.sync_api import Page, expect from pages.base_page import BasePage BASE_URL = "https://the-internet.herokuapp.com" class LoginPage(BasePage): """Page object for the login page.""" # ── URL ──────────────────────────────────────── URL = f"{BASE_URL}/login" # ── Locators — defined as properties ─────────── @property def username_input(self): return self.page.get_by_label("Username") @property def password_input(self): return self.page.get_by_label("Password") @property def login_button(self): return self.page.get_by_role("button", name="Login") @property def flash_message(self): return self.page.locator("#flash") @property def success_flash(self): return self.page.locator("#flash.success") @property def error_flash(self): return self.page.locator("#flash.error") # ── Actions ───────────────────────────────────── def open(self) -> "LoginPage": """Navigate to the login page.""" self.page.goto(self.URL) return self def fill_username(self, username: str) -> "LoginPage": self.username_input.fill(username) return self def fill_password(self, password: str) -> "LoginPage": self.password_input.fill(password) return self def click_login(self) -> "LoginPage": self.login_button.click() return self def login(self, username: str, password: str) -> "LoginPage": """Fill credentials and click Login — one-step action.""" return (self .fill_username(username) .fill_password(password) .click_login()) # ── Assertions ────────────────────────────────── def expect_success(self) -> "LoginPage": """Assert login succeeded.""" expect(self.success_flash).to_be_visible() expect(self.page).to_have_url(f"{BASE_URL}/secure") return self def expect_error(self, message: str) -> "LoginPage": """Assert login failed with specific message.""" expect(self.error_flash).to_be_visible() expect(self.error_flash).to_contain_text(message) return self def expect_on_login_page(self) -> "LoginPage": """Assert we are still on the login page.""" expect(self.page).to_have_url(self.URL) return self
Locators as @property means each access creates a fresh locator — Playwright locators are lazy, so this is correct and efficient. If the locator string changes, you update one property, and every action and assertion using it updates automatically.

CheckboxesPage — a second page object
Once you have written LoginPage the pattern is clear. Every new page follows the same structure — inherit BasePage, define locators as properties, write action methods, write assertion methods.
Create pages/checkboxes_page.py
# pages/checkboxes_page.py from playwright.sync_api import Page, expect from pages.base_page import BasePage BASE_URL = "https://the-internet.herokuapp.com" class CheckboxesPage(BasePage): """Page object for the checkboxes page.""" URL = f"{BASE_URL}/checkboxes" # ── Locators ───────────────────────────────── def checkbox(self, index: int): """Return checkbox at given index (0 or 1).""" return self.page.locator("input[type='checkbox']").nth(index) @property def first_checkbox(self): return self.checkbox(0) @property def second_checkbox(self): return self.checkbox(1) # ── Actions ────────────────────────────────── def open(self) -> "CheckboxesPage": self.page.goto(self.URL) return self def check_first(self) -> "CheckboxesPage": self.first_checkbox.check() return self def uncheck_second(self) -> "CheckboxesPage": self.second_checkbox.uncheck() return self # ── Assertions ─────────────────────────────── def expect_first_unchecked(self) -> "CheckboxesPage": expect(self.first_checkbox).not_to_be_checked() return self def expect_first_checked(self) -> "CheckboxesPage": expect(self.first_checkbox).to_be_checked() return self def expect_second_checked(self) -> "CheckboxesPage": expect(self.second_checkbox).to_be_checked() return self def expect_second_unchecked(self) -> "CheckboxesPage": expect(self.second_checkbox).not_to_be_checked() return self

Update conftest.py — fixtures that return page objects
Fixtures now instantiate page objects instead of returning raw Playwright pages. Tests receive a LoginPage object — not a bare page. This is the bridge between fixtures and POM.
Add these fixtures to conftest.py
# conftest.py — add these POM fixtures import pytest from playwright.sync_api import Page from pages.login_page import LoginPage from pages.checkboxes_page import CheckboxesPage # ── POM fixtures ───────────────────────────────── @pytest.fixture def login_page_pom(page: Page) -> LoginPage: """Returns a LoginPage object navigated to the login URL.""" lp = LoginPage(page) lp.open() yield lp @pytest.fixture def logged_in_pom(page: Page) -> LoginPage: """Returns LoginPage already logged in.""" lp = LoginPage(page) lp.open().login("tomsmith", "SuperSecretPassword!") yield lp @pytest.fixture def checkboxes_pom(page: Page) -> CheckboxesPage: """Returns a CheckboxesPage object navigated to checkboxes.""" cp = CheckboxesPage(page) cp.open() yield cp
Fixture names end in _pom to distinguish from your existing raw-page fixtures. Keep both — you can migrate tests gradually. Real projects do the same — POM and raw tests coexist during transition.

Tests using POM — see how clean they become
This is the payoff. Tests with POM read like plain English. No locators, no Playwright calls — just actions and assertions expressed through the page object API.
Create test_login_pom.py
# test_login_pom.py import pytest from pages.login_page import LoginPage def test_valid_login(login_page_pom: LoginPage): (login_page_pom .login("tomsmith", "SuperSecretPassword!") .expect_success()) def test_wrong_username(login_page_pom: LoginPage): (login_page_pom .login("wronguser", "wrongpass") .expect_error("Your username is invalid") .expect_on_login_page()) def test_wrong_password(login_page_pom: LoginPage): (login_page_pom .login("tomsmith", "wrongpass") .expect_error("Your password is invalid") .expect_on_login_page()) def test_empty_fields(login_page_pom: LoginPage): (login_page_pom .login("", "") .expect_error("Your username is invalid")) # POM + parametrize together @pytest.mark.parametrize("username, password, error", [ ("wronguser", "pass", "Your username is invalid"), ("tomsmith", "wrong", "Your password is invalid"), ("", "", "Your username is invalid"), ], ids=["bad_user", "bad_pass", "empty"]) def test_invalid_logins_parametrized(login_page_pom: LoginPage, username, password, error): (login_page_pom .login(username, password) .expect_error(error))
Create test_checkboxes_pom.py
# test_checkboxes_pom.py from pages.checkboxes_page import CheckboxesPage def test_initial_state(checkboxes_pom: CheckboxesPage): (checkboxes_pom .expect_first_unchecked() .expect_second_checked()) def test_check_first(checkboxes_pom: CheckboxesPage): (checkboxes_pom .check_first() .expect_first_checked()) def test_uncheck_second(checkboxes_pom: CheckboxesPage): (checkboxes_pom .uncheck_second() .expect_second_unchecked())
pytest test_login_pom.py test_checkboxes_pom.py -v --headed
Look at how the test reads: login_page_pom.login("wronguser", "wrongpass").expect_error("Your username is invalid").expect_on_login_page() — that is a complete test scenario in one chained sentence. Any non-technical stakeholder can read it and understand what it tests.

Why POM is asked in every SDET interview interview critical
Every SDET interview asks "explain Page Object Model". Not because it is hard — because it shows you think about maintainability, not just making tests pass. Here is how to answer it.
The 3 things POM gives you
# 1. MAINTAINABILITY # If the Username label changes to "Email": # Without POM — update 20 test files page.get_by_label("Email").fill(...) # × 20 tests # With POM — update one property in LoginPage @property def username_input(self): return self.page.get_by_label("Email") # one change # 2. READABILITY — tests read like requirements login_page.login(user, pw).expect_success() # vs page.get_by_label("Username").fill(user) page.get_by_label("Password").fill(pw) page.get_by_role("button", name="Login").click() expect(page.locator("#flash.success")).to_be_visible() # 3. REUSABILITY — one page object, many test files from pages.login_page import LoginPage # Used in: test_login.py, test_security.py, test_session.py...
How to answer in an interview
# "What is Page Object Model?" # Answer: # POM is a design pattern where each page of the application # has a corresponding class that encapsulates all the locators, # actions, and assertions for that page. # Tests call methods on page objects instead of using # Playwright/Selenium directly. # Benefits: # - Maintainability: locator changes = one class update # - Readability: tests read like requirements # - Reusability: page objects shared across test files # - Separation of concerns: test logic vs page interactions
When the interviewer says "show me an example", say: "I have a LoginPage class with an open() method, a login(username, password) method, and expect_success() / expect_error() assertion methods. Tests chain these methods — login_page.open().login(user, pw).expect_success()." That answer gets you to the next round.

Day 20 challenge — build a DropdownPage
You have LoginPage and CheckboxesPage. Now build DropdownPage from scratch following the same pattern.
Build pages/dropdown_page.py + tests
Part 1 — pages/dropdown_page.py
Inherit from BasePage. Add:
• URL class variable pointing to /dropdown
• dropdown property — locator for #dropdown
• open() — navigate to URL
• select_option(value) — select by value, return self
• expect_value(value) — assert to_have_value(), return self
• expect_text(text) — assert to_contain_text(), return self

Part 2 — conftest.py
Add a dropdown_pom fixture that returns a DropdownPage already navigated.

Part 3 — test_dropdown_pom.py
Write 3 tests using method chaining:
• test_default_visible — assert dropdown is visible
• test_select_option1 — select "1", assert value and text
• test_select_option2 — select "2", assert value and text

Bonus — add parametrize
Combine dropdown_pom fixture with parametrize for both options in one test function.
When this is done you have a complete POM framework with 3 page objects, a BasePage, POM fixtures in conftest, and tests using method chaining. Push this to GitHub — it is Portfolio Project #1 complete.
