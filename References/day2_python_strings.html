Day 2 Python — strings and f-strings in depth for QA automation

What is a string — and why QA engineers use them constantly
A string is any text wrapped in quotes. In test automation, almost everything visible on screen is a string — page titles, error messages, URLs, button labels, form values, API response fields.

When you write expect(page).to_have_title("Dashboard") in Playwright — that "Dashboard" is a string. When you assert an API response contains "success" — that is a string. Mastering strings means mastering assertions.
Three ways to create a string — all valid
# Single quotes url = 'https://staging.myapp.com' # Double quotes (most common) message = "You logged into a secure area!" # Triple quotes — for long text spanning multiple lines test_description = """This test verifies that a valid user can log in and see the dashboard.""" print(url) print(message) print(test_description)
Output
https://staging.myapp.com
You logged into a secure area!
This test verifies that a valid user
can log in and see the dashboard.
Use double quotes as your default. Use single quotes when your string contains an apostrophe: 'user's profile' — or just escape it: "user\'s profile". Pick one style and be consistent throughout a project.
Strings have a length — check it with len()
password = "SuperSecretPassword!" print(len(password)) # 20 # Useful in QA: check password meets minimum length requirement min_length = 8 print(len(password) >= min_length) # True
Output
20
True

f-strings — the most important string skill use every day
An f-string lets you embed variables directly inside a string. Put f before the opening quote, then wrap any variable in {curly braces}. Python replaces the braces with the variable's value at runtime.

You will use f-strings in every test file you ever write — for building URLs, printing test results, writing error messages, and logging what your test is doing.
Type this into day2.py
# Without f-string — clunky env = "staging" path = "/login" url = "https://" + env + .myapp.com" + path # messy # With f-string — clean url = f"https://{env}.myapp.com{path}" print(url) # https://staging.myapp.com/login # Any variable type works inside braces test_num = 7 passed = True print(f"Test {test_num} result: {passed}") # Expressions work inside braces too total = 42 fails = 4 print(f"Pass rate: {total - fails}/{total}") print(f"Percentage: {round((total-fails)/total*100, 1)}%")
Output
https://staging.myapp.com/login
Test 7 result: True
Pass rate: 38/42
Percentage: 90.5%
The QA logging pattern — use this in every test
username = "tomsmith" env = "staging" browser = "chromium" status = 200 # Print what your test is about to do — before the action print(f"[TEST] Logging in as '{username}' on {env} using {browser}") # Print what happened — after the action print(f"[RESULT] API responded with status: {status}") print(f"[PASS] Expected 200, got {status} — {'PASS' if status == 200 else 'FAIL'}")
Output
[TEST] Logging in as 'tomsmith' on staging using chromium
[RESULT] API responded with status: 200
[PASS] Expected 200, got 200 — PASS
The most common f-string mistake: forgetting the f before the quote. "Hello {name}" prints literally as Hello {name} — the braces are not replaced. Always check you wrote f"Hello {name}".

String methods — built-in tools on every string
A method is a function that belongs to a string. You call it with a dot: my_string.method(). These are the ones you will actually use in QA automation — not all 40+ Python string methods, just the ones that matter.
.lower()
Convert to all lowercase — for case-insensitive comparisons
"ERROR".lower() → "error"
.upper()
Convert to all uppercase
"pass".upper() → "PASS"
.strip()
Remove spaces from both ends — clean scraped text
" hello ".strip() → "hello"
.replace(a,b)
Replace all occurrences of a with b
"prod.app.com".replace("prod","staging")
.startswith()
Returns True if string starts with the given text
url.startswith("https") → True/False
.endswith()
Returns True if string ends with the given text
file.endswith(".png") → True/False
.split(sep)
Split string into a list by separator
"a,b,c".split(",") → ["a","b","c"]
.count()
Count how many times a substring appears
"aababc".count("a") → 3
"x" in s
True if x is found anywhere in s
"error" in message → True/False
.find()
Returns index of first match, or -1 if not found
"hello".find("ll") → 2
Type all of these — see each one in action
message = " Your Username Is Invalid! " # Clean it first clean = message.strip() print(clean) # "Your Username Is Invalid!" # Lowercase for safe comparison print(clean.lower()) # "your username is invalid!" # Does it mention "invalid"? — case-insensitive check print("invalid" in clean.lower()) # True print("success" in clean.lower()) # False url = "https://staging.myapp.com/login" print(url.startswith("https")) # True print(url.endswith("/login")) # True print(url.replace("staging", "prod")) # https://prod.myapp.com/login # Split a URL into parts parts = url.split("/") print(parts) # ['https:', '', 'staging.myapp.com', 'login']
Output
Your Username Is Invalid!
your username is invalid!
True
False
True
True
https://prod.myapp.com/login
['https:', '', 'staging.myapp.com', 'login']
Methods can be chained: message.strip().lower() — strip spaces first, then lowercase. Read it left to right. This is the most common pattern when comparing text scraped from a browser, because scraped text often has extra spaces and unexpected capitalisation.

QA Example 1 — Dynamic URL builder f-strings
In every project you'll have a base URL and dozens of paths. Instead of hardcoding the full URL everywhere, build them dynamically. Change env once at the top and all URLs update.
Create day2_ex1.py and type this
# Central config — change env here, everything updates env = "staging" protocol = "https" domain = "myapp.com" # Build URLs for every page you'll test base_url = f"{protocol}://{env}.{domain}" login_url = f"{base_url}/login" dashboard_url = f"{base_url}/dashboard" profile_url = f"{base_url}/profile" api_url = f"{base_url}/api/v1" print(f"Base: {base_url}") print(f"Login: {login_url}") print(f"Dashboard: {dashboard_url}") print(f"Profile: {profile_url}") print(f"API: {api_url}") # Now switch to prod — change ONE line env = "prod" base_url = f"{protocol}://{env}.{domain}" login_url = f"{base_url}/login" print(f"\nSwitched to prod: {login_url}")

QA Example 2 — Error message validator .strip() .lower() in
When you scrape text from a browser, it often comes back with extra spaces, random capitalisation, or newlines. Before asserting, you clean and normalise it. This is the single most used string pattern in UI test automation.
Create day2_ex2.py
# Simulating text scraped from a web page # Real Playwright: text = page.locator("#flash").inner_text() scraped_text = " \n Your username is invalid! \n " # Step 1: clean it clean = scraped_text.strip() print(f"After strip: '{clean}'") # Step 2: normalise case normalised = clean.lower() print(f"Normalised: '{normalised}'") # Step 3: assert — does it contain expected text? expected = "invalid" print(f"Contains '{expected}': {expected in normalised}") # Real assertion pattern — one chained line assert "invalid" in scraped_text.strip().lower(), \ f"Expected error message to contain 'invalid', got: '{clean}'" print("Assertion passed!") # Now test an exact match expected_exact = "Your username is invalid!" print(f"Exact match: {clean == expected_exact}") # True
Output
After strip: 'Your username is invalid!'
Normalised: 'your username is invalid!'
Contains 'invalid': True
Assertion passed!
Exact match: True
In Playwright you will write exactly this: assert "invalid" in page.locator("#flash").inner_text().strip().lower() — the same chain you just practiced. You're already writing real assertion logic.

QA Example 3 — URL security validator .startswith() .endswith() in
A very common QA task: verify a redirect went to the right URL, or that the app never serves pages over plain HTTP. You'll write this kind of check in Playwright after clicks and form submissions.
Create day2_ex3.py
# URLs returned after various actions in the app urls = [ "https://staging.myapp.com/dashboard", "http://staging.myapp.com/login", # insecure! "https://staging.myapp.com/profile", "https://evil.com/phishing", # wrong domain! "https://staging.myapp.com/logout", ] print("--- URL Security Check ---") for url in urls: is_https = url.startswith("https") is_domain = "myapp.com" in url is_safe = is_https and is_domain status = "PASS" if is_safe else "FAIL" print(f"[{status}] {url}") if not is_https: print(f" ^ Not HTTPS!") if not is_domain: print(f" ^ Wrong domain!")
Output
[PASS] https://staging.myapp.com/dashboard
[FAIL] http://staging.myapp.com/login
^ Not HTTPS!
[PASS] https://staging.myapp.com/profile
[FAIL] https://evil.com/phishing
^ Wrong domain!
[PASS] https://staging.myapp.com/logout
You just used a loop (from tomorrow's lesson) — don't worry about fully understanding it yet. Focus on what the string methods are doing: startswith() checks the beginning, in checks anywhere inside. That logic is what matters.

QA Example 4 — Test result report formatter f-strings advanced
f-strings can do more than just insert values — they can format numbers, align text, and even contain inline conditions. This is exactly the kind of output you'll print at the end of a test run.
Create day2_ex4.py
# Test session data app_name = "ShopCart Pro" env = "staging" browser = "chromium" total = 47 passed = 43 failed = 4 duration_s = 127.5 # Calculate derived values pass_rate = (passed / total) * 100 skipped = total - passed - failed # f-string formatting tricks # :.1f = 1 decimal place for floats # :>6 = right-align in 6 characters wide print(f"{'=' * 40}") print(f" Test Report — {app_name}") print(f"{'=' * 40}") print(f" Environment : {env}") print(f" Browser : {browser}") print(f" Duration : {duration_s:.1f}s") print(f"{'─' * 40}") print(f" Total tests : {total}") print(f" Passed : {passed}") print(f" Failed : {failed}") print(f" Skipped : {skipped}") print(f" Pass rate : {pass_rate:.1f}%") print(f"{'=' * 40}") print(f" Status: {'PASSED' if failed == 0 else 'FAILED'}")
Output
========================================
Test Report — ShopCart Pro
========================================
Environment : staging
Browser : chromium
Duration : 127.5s
────────────────────────────────────────
Total tests : 47
Passed : 43
Failed : 4
Skipped : 0
Pass rate : 91.5%
========================================
Status: FAILED
:.1f inside braces formats a float to 1 decimal place. "=" * 40 repeats the = character 40 times. 'PASSED' if failed == 0 else 'FAILED' is an inline condition — picks one string or the other based on the value of failed.

QA Example 5 — API response field checker strings in real tests
When an API returns JSON, the field values are strings. You'll check them constantly — email format, status field value, error message text. This example previews exactly what your Week 4 API tests will look like.
Create day2_ex5.py
# Simulating a JSON response from reqres.in/api/users/2 # Real code: data = response.json()["data"] first_name = "Janet" last_name = "Weaver" email = "janet.weaver@reqres.in" avatar_url = "https://reqres.in/img/faces/2-image.jpg" # Build full name from parts full_name = f"{first_name} {last_name}" print(f"Full name: {full_name}") # Validate email format (basic checks) has_at = "@" in email has_dot = "." in email.split("@")[-1] not_empty = len(email.strip()) > 0 looks_valid = has_at and has_dot and not_empty print(f"Email: {email}") print(f" Has @: {has_at}") print(f" Has dot: {has_dot}") print(f" Not empty: {not_empty}") print(f" Looks valid: {looks_valid}") # Validate avatar URL print(f"\nAvatar URL checks:") print(f" Is HTTPS: {avatar_url.startswith('https')}") print(f" Is image: {avatar_url.endswith(('.jpg','.png','.webp'))}") print(f" Has user ID: {'2' in avatar_url}") # Domain extraction from email domain = email.split("@")[1] print(f"\nEmail domain: {domain}") print(f"Domain is reqres.in: {domain == 'reqres.in'}")
Output
Full name: Janet Weaver
Email: janet.weaver@reqres.in
Has @: True
Has dot: True
Not empty: True
Looks valid: True
 
Avatar URL checks:
Is HTTPS: True
Is image: True
Has user ID: True
 
Email domain: reqres.in
Domain is reqres.in: True
.split("@")[1] splits the email on @ and takes the second part (index 1 = everything after @). This is real parsing logic you will use in API tests when checking email fields in JSON responses.
