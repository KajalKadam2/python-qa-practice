The full 6-week arc
Week 1 — Days 1–7~2h/day
Python from zero. Variables, strings, f-strings, lists, dicts, conditions, loops. Every concept practiced with QA examples — not abstract CS exercises. End goal: read and write basic Python without googling every line.
Week 2 — Days 8–14~2h/day
Python functions + OOP + Git. Functions, default args, return values, classes, POM pattern, self, inheritance. Plus Git basics — commit, push, GitHub repo. End goal: write a LoginPage class and push it to GitHub.
Week 3 — Days 15–21~2.5h/day
Playwright from scratch. Installation, first test, locators, assertions, fixtures, conftest, parametrize, Page Object Model applied. Practice on the-internet.herokuapp.com daily. End goal: 10+ working Playwright tests using POM.
Week 4 — Days 22–28~2.5h/day
API testing + Postman + SQL basics. REST concepts, Postman, Python requests library, assert JSON responses, status codes. Basic SQL SELECT/WHERE/JOIN. End goal: 10+ API tests on a live practice API, pushed to GitHub.
Week 5 — Days 29–35~3h/day
CI/CD + portfolio build + Selenium awareness. GitHub Actions pipeline, 3 clean GitHub repos, resume rewrite, LinkedIn update. Selenium basics — enough to answer interview questions. End goal: portfolio live, applications start going out.
Week 6 — Days 36–45~2h/day + applying
Interview prep + active job hunt. Top 25 SDET questions, mock interviews, coding round practice, scenario answers using your 7.4yr QA experience. 5+ applications per day. End goal: first interviews booked.
You have 7.4 years of QA experience. That's not a gap — it's a shortcut. You already know WHAT to test. You're only learning HOW to express it in code. Most fresh automation learners don't have that instinct. You do.
Week 1 — Days 1–7
Python from zero
Focus 100% on Python. No Playwright yet. Build the language muscle first.
Daily time — 2 hours
60 min — new concept + read examples
50 min — write it yourself from scratch
10 min — review what you wrote, fix errors
Day 12h
Setup + variables. Install Python, VS Code, run your first print("Hello"). Learn: variables, integers, strings, booleans, type(). Write 10 lines: store a URL, a username, a timeout value, a True/False flag. Just get comfortable running Python.
Day 22h
Strings + f-strings. This is the most-used thing in test automation. Learn: f"text {variable}", .strip(), .lower(), .startswith(), in keyword. Exercise: build 5 dynamic test URLs using f-strings with different env names (staging/prod/dev).
Day 32h
Lists. Create, access by index, loop over, append(), len(), negative index. Exercise: make a list of 5 invalid emails. Print each one with a message like "Testing invalid input: noatsign". Make a list of browsers and print "Running on: chromium" for each.
Day 42h
Dictionaries. Key-value pairs, access with dict["key"], safe access with .get(), loop with .items(). Exercise: create a test_user dict with username, password, role. Create a config dict with base_url, timeout, headless. Print values using f-strings.
Day 52h
if / elif / else. Conditions, comparison operators (==, !=, >), in for membership, is None. Exercise: write logic that checks an HTTP status code and prints "Pass", "Not Found", or "Server Error". Check if a role is in an allowed list.
Day 62h
for loops + range(). Loop over a list, loop with enumerate(), loop over dict items, range(3). Exercise: loop over your invalid emails list and print each. Loop over a config dict and print every key=value pair. Use range to print "Retry 1", "Retry 2", "Retry 3".
Day 7 — Consolidation2h
Mini project — no guidance. Write a Python script that: stores 3 test users as dicts in a list, loops over them, prints a formatted login attempt message for each one, checks if each user's role is "admin" or not, and prints PASS/FAIL accordingly. This uses every concept from the week.
Week 1 checkpoint — can you do these without looking?
Create a dict with 4 keys and print each value with an f-string
Loop over a list and print each item with its index number
Write an if/elif/else that checks a status code
Build a URL string dynamically using f-strings and a variable
Week 2 — Days 8–14
Python functions + OOP + Git
Build reusable code. Understand classes. Push everything to GitHub.
Daily time — 2 hours
50 min — new concept
60 min — hands-on coding
10 min — push to GitHub
Day 82h
Functions — basics. def, parameters, return. Exercise: write build_url(env, path) that returns a full URL. Write get_full_name(first, last). Write is_valid_status(code) that returns True if code is 200–299.
Day 92h
Default arguments + docstrings. def login(user="tomsmith", pw="Secret"). Callers can override or use defaults. Write a docstring for every function. Exercise: write a login() function with default credentials, override for negative test cases.
Day 102h
try / except. Handle errors without crashing. try / except Exception as e / finally. Exercise: write safe_divide(a, b) that handles divide-by-zero. Write read_config(file) that handles file-not-found gracefully with a message.
Day 112h
Classes + self + __init__. This is the big one. Create a TestUser class with username, password, role attributes. Create 3 instances. Print their details. Add a method get_credentials() that returns a formatted string.
Day 122h
Inheritance. class AdminUser(TestUser). Child gets all parent methods free. Add extra methods only the child has. Exercise: create a BasePage class with a get_title() method. Create LoginPage(BasePage) that adds a login() method. This IS the Page Object Model pattern.
Day 132h
Git from zero. Install Git. Learn: git init, git add ., git commit -m "message", git push. Create a GitHub account. Create your first repo "python-qa-practice". Push all your week 1 and 2 scripts there.
Day 14 — Mini project2h
Build a TestUser class system. BasePerson class → TestUser(BasePerson) child with login method → AdminUser(TestUser) child with extra permissions. Create instances, call methods, print output. Push to GitHub with a README that explains what it does.
Week 2 checkpoint
Write a function with default args and return value — from memory
Create a class with __init__, 2 attributes, and 2 methods
Create a child class that inherits from a parent
Your GitHub has at least 1 repo with your code pushed
Week 3 — Days 15–21
Playwright from scratch
Now Python clicks. Apply it directly to Playwright. Every day on the-internet.herokuapp.com.
Daily time — 2.5 hours
30 min — new Playwright concept
90 min — write + run real tests
30 min — fix failures, push to GitHub
Day 152.5h
Install + first test. pip install pytest-playwright, playwright install. Write your very first test: go to the-internet.herokuapp.com, assert page title. Run with pytest --headed and WATCH it run. This moment matters — see your code control a browser.
Day 162.5h
Locators. Practice all 5 locator types on the practice site: get_by_role, get_by_label, get_by_text, get_by_placeholder, locator("css"). Write 1 test using each. Feel which ones are readable and which are fragile.
Day 172.5h
Assertions + actions. expect().to_be_visible(), to_have_text(), to_have_title(), to_be_enabled(). Write login test (happy path), then negative test with wrong credentials. Assert the success message AND the error message.
Day 182.5h
pytest fixtures + conftest.py. Create a conftest.py with a fixture that navigates to login page before each test. Use it in 3 tests. Understand @pytest.fixture and why it removes repeated page.goto() from every test.
Day 192.5h
parametrize. @pytest.mark.parametrize to test 5 invalid login combos in one test function. This is where your Python loops knowledge pays off — parametrize IS a loop over test data. Write it, run it, watch 5 tests run from 1 function.
Day 202.5h
Page Object Model in Playwright. Create pages/login_page.py — a real LoginPage class using what you learned in Week 2. Move all locators and actions into the class. Rewrite your tests to use LoginPage(page).login(user, pw). Feel how clean the tests become.
Day 21 — Portfolio build2.5h
GitHub repo: "playwright-ui-tests". Folder structure: pages/, tests/, conftest.py. Write 10 tests total across: login, checkboxes, dropdowns, forms. Run all tests clean with pytest -v. Write a README. Push. This is Portfolio Project #1.
Week 3 checkpoint
10+ Playwright tests running clean on pytest
LoginPage class using Page Object Model
conftest.py with at least one fixture
parametrize used in at least one test
Repo pushed to GitHub with proper folder structure
Week 4 — Days 22–28
API testing + Postman + SQL basics
APIs are in every job description. This week makes you a full-stack QA tester.
Daily time — 2.5 hours
40 min — concept + theory
80 min — hands-on testing
30 min — write it as Python tests
Day 222.5h
REST API concepts. What is an API, HTTP methods (GET/POST/PUT/DELETE), status codes (200/201/400/401/404/500), headers, JSON. No coding today — just understand it. Open Postman, make a GET call to reqres.in/api/users. Read the JSON response. This is what you'll be asserting.
Day 232.5h
Postman hands-on. GET list of users, GET single user, POST create user, DELETE user — all on reqres.in. Write Postman tests (the Tests tab) asserting status codes and response fields. Run the collection. Postman is a tool you can already put on your resume after today.
Day 242.5h
Python requests library. import requests. Write: response = requests.get(url), assert response.status_code == 200, parse response.json(). Mirror your Postman calls from Day 23 but now in Python code.
Day 252.5h
API tests with pytest. Wrap your requests calls in proper def test_ functions. Assert nested JSON fields: data["data"]["email"] == "janet.weaver@reqres.in". Test POST: send a body, assert the response contains what you sent. 5 tests today.
Day 262.5h
API test suite — build it out. Write 10 tests total: GET list, GET single, POST create, PUT update, DELETE, 404 not found, 400 bad request. Use parametrize for testing multiple user IDs. Push to new GitHub repo "api-testing-python". Portfolio Project #2 done.
Day 272.5h
SQL basics. Practice on sqlzoo.net (free, browser-based). Learn: SELECT * FROM users, WHERE email = 'x', COUNT(*), JOIN. Goal: in an interview, answer "how would you verify data in the database?" confidently.
Day 28 — Consolidation2.5h
Review + Selenium awareness. Spend 1h reviewing your API tests — are they clean, readable, well-named? Spend 1h installing Selenium and writing the same login test you wrote in Playwright. Just feel the difference. You don't need to master it — just know it exists and how it differs.
Week 4 checkpoint
10+ API tests in Python using requests + pytest — live on GitHub
Can explain GET vs POST vs PUT vs DELETE in an interview
Can write a basic SQL SELECT with WHERE clause
Have run at least 1 Selenium test — can compare both frameworks
Week 5 — Days 29–35
CI/CD + portfolio polish + start applying
Make your work visible. A pipeline badge on GitHub is worth more than 10 certifications.
Daily time — 3 hours (push harder this week)
60 min — technical work
60 min — portfolio + resume
60 min — job applications
Day 29–303h each
GitHub Actions. Create .github/workflows/tests.yml in your Playwright repo. Make tests run automatically on every push. When it goes green, add the badge to your README: ![Tests](workflow badge URL). This is the single most impressive thing you can show a hiring manager.
Day 313h
Portfolio Project #3 — combined test suite. One repo that tests both UI (Playwright) AND API (requests) for the same user flow: register via API → verify via UI → delete via API. Shows you think end-to-end, not just "I click buttons".
Day 323h
Resume rewrite. Headline: "QA Automation Engineer | Python | Playwright | 7+ Years QA". Skills section: Python, Playwright, pytest, REST API Testing, Postman, GitHub Actions, Git, SQL, Selenium (basics), Page Object Model. Link your GitHub prominently. Quantify: "Built 30+ automated tests covering UI and API layers".
Day 333h
LinkedIn overhaul + first applications. Headline = "QA Automation Engineer | Playwright | Python | Open to Work". About section reframes your 7.4 years. Add your GitHub. Set Open to Work. Send your first 5 applications. Don't wait for perfect — done beats perfect here.
Day 34–353h each
Docker basics + polish. Pull the Playwright Docker image. Run your tests inside a container. You don't need to master Docker — just enough to say "yes, I've containerised my tests" in an interview. Spend remaining time polishing your 3 repos: clean code, good READMEs, all tests passing.
By end of Day 35 you have: 3 GitHub repos with CI/CD badges, a rewritten resume, an updated LinkedIn, and 5+ applications sent. You are officially job hunting.
Week 6 — Days 36–45
Interview sprint + active applying
Your 7.4 years is your weapon here. Connect every answer to your real experience.
Daily time — 2h prep + 1h applying (every single day)
Day 36–373h each
Top 25 SDET interview questions. Write your answer to each one. Don't just think it — type it out. For technical questions: explain the concept + give an example from your code. For QA scenarios: use your real experience and add the automation angle. "In my manual QA years I noticed X. Now I automate that check by..."
Day 383h
Coding round prep. 32% of SDET interviews now include coding tests. Practice on HackerRank: write a function, fix a bug in code, write a class. Easy level only. 3 problems per day. Focus on: string manipulation, list operations, basic OOP — all things you now know.
Day 393h
Mock interview — say answers OUT LOUD. Ask me to interview you right now. Or record yourself on video answering 10 questions. Watching yourself back is brutal and effective. Filler words, unclear explanations, and gaps in knowledge show up immediately.
Day 40–423h each
Apply aggressively. 5+ applications per day. Search: "QA Automation Engineer", "SDET", "Automation Test Engineer", "Software Engineer in Test", "QA Engineer Playwright". Tailor just 2 lines of cover for each company's stack. Follow up on earlier applications after 5 days.
Day 43–453h each
Interview-week mindset. By now you may have calls booked. Review your own repos before each call — be ready to walk through your code live. Keep applying even while interviewing. Your goal is multiple offers, not just one.
Your advantage in every interview
7.4 years means you know test strategy, edge cases, regression thinking, bug reporting, stakeholder communication. Most junior automators only know how to run scripts. You know WHY the tests exist.
