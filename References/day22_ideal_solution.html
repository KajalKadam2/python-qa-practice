Schema validation — check every record, not just the first. A nested loop checks every user has every required field. The assertion message includes the user ID and field name so when it fails in CI you know exactly which user and which field without reading the JSON. This is the difference between a useful failure message and a useless one.
assert body["total"] == 12 assert body["per_page"] == 6 assert len(body["data"]) == 6
Assert pagination metadata, not just the data. A status 200 with correct data but wrong pagination metadata is still a bug. total, per_page, and page tell clients how to paginate — if they are wrong, mobile apps show incorrect page counts. Your QA instinct should flag pagination fields as critical.
Test 2 Create user — type check on id field
assert isinstance(body["id"], str), ( f"Expected id to be str, got {type(body['id']).__name__}" )
Type assertions catch contract violations. reqres returns id as a string — not an integer. A client that stores it as int and sends it back would cause mismatches. isinstance(value, str) asserts the type explicitly. The message includes the actual type so failures are immediately actionable.

The 5 mistakes most people made
Mistake 1 — class variable name mismatch (BASE_URL vs BASE)
# Wrong — defined as BASE_URL but used as self.BASE class ReqresAPI: BASE_URL = "https://reqres.in/api" # defined def get_user(self, user_id): return requests.get(f"{self.BASE}/...") # AttributeError! # Correct — name must match exactly class ReqresAPI: BASE = "https://reqres.in/api" # defined as BASE def get_user(self, user_id): return requests.get(f"{self.BASE}/...") # matches
Python is case-sensitive and exact about attribute names. self.BASE_URL and self.BASE are two different attributes. Define it once, use the exact same name everywhere. This was the error that caused all 3 test failures in your run.
Mistake 2 — using requests.get for POST/PATCH/DELETE
# Wrong — all methods using GET def create_user(self, name, job): return requests.get(...) # will never return 201 def update_user(self, id, **kwargs): return requests.get(...) # will never return 200 with updatedAt # Correct — each method uses the right HTTP verb def create_user(self, name, job): return requests.post(...) def update_user(self, id, **kwargs): return requests.patch(...)
HTTP method matters — the API routes differently based on GET/POST/PATCH/DELETE. A GET to /users returns a list. A POST to /users creates a user. Same URL, completely different behaviour. Using the wrong method is like knocking on the wrong door.
Mistake 3 — only asserting status code, not body fields
# Incomplete — only checks the door opened, not what's inside def test_create_and_verify_user(): r = requests.post(...) assert r.status_code == 201 # that's it # Complete — verifies the actual data def test_create_and_verify_user(): r = requests.post(...) assert r.status_code == 201 body = r.json() assert body["name"] == "Kajal Kadam" assert body["job"] == "QA Engineer" assert "id" in body assert isinstance(body["id"], str)
Status code 201 only means the server accepted the request — not that it processed it correctly. An API could return 201 with the wrong name or no ID. Body assertions verify the actual contract. Your 7.4 years of QA experience knows this — a green status light does not mean the feature works.
Mistake 4 — not adding HEADERS to every request
# Missing headers — gets 401 r = requests.get(f"{BASE_URL}/users/2") # Correct — headers on every call r = requests.get(f"{BASE_URL}/users/2", headers=HEADERS)
reqres.in now requires the API key on every request. Define HEADERS once at the top of the file and pass it to every request. Missing it on even one request causes a 401 that looks like a test failure but is actually a configuration issue.
Mistake 5 — calling .json() on a 204 DELETE response
# Wrong — 204 has no body, .json() raises an error r = requests.delete(f"{BASE_URL}/users/2", headers=HEADERS) body = r.json() # JSONDecodeError — no content! # Correct — just assert the status code and empty text r = requests.delete(f"{BASE_URL}/users/2", headers=HEADERS) assert r.status_code == 204 assert r.text == ""
204 No Content means the response body is intentionally empty. Calling .json() on it raises a JSONDecodeError. For DELETE tests: assert status 204, optionally assert r.text == "". Never call .json() on a 204 response.
