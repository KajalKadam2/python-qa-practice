Day 19 — Assertions and waits in depth
On Day 15 you used expect(element).to_be_visible() and it worked. But production websites have elements that appear after API calls, animations, or user actions. Knowing which assertion to use and how Playwright's auto-waiting works is what separates flaky tests from reliable ones.

Today you learn every assertion you need, how Playwright waits automatically, and when to add explicit waits.
Why tests go flaky — the root cause
# Flaky test — races against the page loading page.click("#submit") # assert runs immediately — element may not exist yet assert page.locator(".result").is_visible() # ← plain Python — no waiting # Reliable — Playwright retries until timeout page.click("#submit") expect(page.locator(".result")).to_be_visible() # ← retries for 5 seconds
The most important rule of Day 19: always use expect(locator).to_be_*() instead of assert locator.is_*(). The expect version retries automatically for 5 seconds. The assert version runs once and fails immediately if the element is not ready yet.
1 of 9
Visibility assertions — the ones you use every day
These four cover 80% of all assertions you will ever write. Know them cold.
to_be_visible()
Element exists AND is visible on screen
to_be_hidden()
Element is hidden or does not exist
to_be_enabled()
Form element is not disabled
to_be_disabled()
Form element is disabled
Create Day19_assertions.py
from playwright.sync_api import Page, expect BASE = "https://the-internet.herokuapp.com" def test_visibility_assertions(page: Page): page.goto(f"{BASE}/login") # Element is visible on screen expect(page.get_by_role("button", name="Login")).to_be_visible() # Element is enabled (not greyed out) expect(page.get_by_label("Username")).to_be_enabled() # NOT assertions — the negative form expect(page.locator("#flash")).to_be_hidden() # flash not shown yet def test_visible_after_action(page: Page): page.goto(f"{BASE}/login") # Flash is hidden before login expect(page.locator("#flash")).to_be_hidden() # After failed login — flash appears page.get_by_label("Username").fill("wrong") page.get_by_label("Password").fill("wrong") page.get_by_role("button", name="Login").click() # Now flash is visible — expect retries until it appears expect(page.locator("#flash.error")).to_be_visible()
pytest Day19_assertions.py -v --headed
to_be_hidden() passes if the element is either not in the DOM at all OR exists but is hidden with CSS. It does not care which — both count as hidden. Use it to assert "this error message is not showing yet".
2 of 9
Text and value assertions
After visibility, the next most common assertion type is checking what text or value an element contains. Know the difference between exact and contains matching.
to_have_text()
Exact full text match
to_contain_text()
Text contains the string (partial match)
to_have_value()
Input field value matches exactly
to_have_attribute()
Element has attribute with value
Add to Day19_assertions.py
def test_text_assertions(page: Page): page.goto(f"{BASE}/login") page.get_by_label("Username").fill("wrong") page.get_by_label("Password").fill("wrong") page.get_by_role("button", name="Login").click() flash = page.locator("#flash") # to_contain_text — partial match — most flexible expect(flash).to_contain_text("Your username is invalid") # to_have_text — full exact match including whitespace # Use to_contain_text unless you need the full string def test_value_assertions(page: Page): page.goto(f"{BASE}/login") # Fill field — assert value was set correctly page.get_by_label("Username").fill("tomsmith") expect(page.get_by_label("Username")).to_have_value("tomsmith") # Dropdown value page.goto(f"{BASE}/dropdown") page.locator("#dropdown").select_option("1") expect(page.locator("#dropdown")).to_have_value("1") def test_attribute_assertion(page: Page): page.goto(f"{BASE}/login") # Assert input type attribute expect(page.locator("#username")).to_have_attribute("type", "text") expect(page.locator("#password")).to_have_attribute("type", "password")
to_have_text() does a full exact match — it fails if there is extra whitespace or surrounding text. to_contain_text() is a partial match — it passes if your string appears anywhere in the element. Use to_contain_text for flash messages and dynamic content where extra text may appear.
3 of 9
URL, title and count assertions
These assert the page itself — not a specific element. Essential for navigation tests and dynamic lists.
to_have_url()
Page URL matches exactly
to_have_title()
Page title matches
to_have_count()
Locator matches exactly N elements
to_have_class()
Element has a specific CSS class
Add to Day19_assertions.py
def test_url_assertions(page: Page): page.goto(f"{BASE}/login") expect(page).to_have_url(f"{BASE}/login") # After login — URL changes to /secure page.get_by_label("Username").fill("tomsmith") page.get_by_label("Password").fill("SuperSecretPassword!") page.get_by_role("button", name="Login").click() expect(page).to_have_url(f"{BASE}/secure") def test_count_assertion(page: Page): page.goto(f"{BASE}/checkboxes") expect(page.locator("input[type='checkbox']")).to_have_count(2) def test_dynamic_count(page: Page): page.goto(f"{BASE}/add_remove_elements/") # No delete buttons initially expect(page.get_by_role("button", name="Delete")).to_have_count(0) # Add 2 elements page.get_by_role("button", name="Add Element").click() page.get_by_role("button", name="Add Element").click() expect(page.get_by_role("button", name="Delete")).to_have_count(2) # Remove one — count drops to 1 page.get_by_role("button", name="Delete").first.click() expect(page.get_by_role("button", name="Delete")).to_have_count(1)
to_have_count(0) is how you assert that something is NOT on the page without getting an error. It retries until there are exactly 0 matching elements — perfect for asserting items were deleted or error messages cleared.

Checkbox, focus and state assertions
Form state assertions go beyond visibility — they check whether checkboxes are ticked, fields are focused, and elements are in the right state for the user.
to_be_checked()
Checkbox or radio is checked
not_to_be_checked()
Checkbox or radio is unchecked
to_be_focused()
Element has keyboard focus
to_be_editable()
Input can be typed into
Add to Day19_assertions.py
def test_checkbox_assertions(page: Page): page.goto(f"{BASE}/checkboxes") first = page.locator("input[type='checkbox']").first second = page.locator("input[type='checkbox']").last # Initial state expect(first).not_to_be_checked() expect(second).to_be_checked() # Check first, uncheck second first.check() second.uncheck() # Assert state flipped expect(first).to_be_checked() expect(second).not_to_be_checked() def test_editable_assertion(page: Page): page.goto(f"{BASE}/login") # Inputs should be editable expect(page.get_by_label("Username")).to_be_editable() expect(page.get_by_label("Password")).to_be_editable()

Auto-waiting — how Playwright prevents flaky tests
Playwright automatically waits before every action and every expect assertion. You almost never need explicit waits. Understanding this prevents you from adding unnecessary time.sleep() calls that slow tests down.
What Playwright waits for automatically
# Before EVERY action (click, fill, check, select_option): # ✓ Element exists in DOM # ✓ Element is visible (not hidden) # ✓ Element is enabled (not disabled) # ✓ Element is stable (not animating) # ✓ Element receives events (not covered by overlay) # → Only then does the action execute # Before EVERY expect assertion: # ✓ Retries the condition for up to 5 seconds (default timeout) # ✓ Passes as soon as the condition is true # ✓ Fails only when timeout expires
What this means for your tests
# You DO NOT need this: page.click("#submit") import time time.sleep(2) # ← NEVER do this. Slow and fragile. expect(page.locator(".result")).to_be_visible() # This is all you need — expect handles the wait: page.click("#submit") expect(page.locator(".result")).to_be_visible() # retries for 5s
When you DO need explicit waits
# 1. Waiting for navigation to complete page.click("#submit") page.wait_for_url("**/dashboard") # waits until URL matches pattern # 2. Waiting for an element to appear (when you need the element itself) page.wait_for_selector(".result") # waits for element, returns it # 3. Custom timeout for slow pages expect(page.locator(".slow-chart")).to_be_visible(timeout=15000) # 15s # 4. Waiting for network requests to finish page.wait_for_load_state("networkidle") # all network activity stopped
Never use time.sleep() in tests. It makes every test run slower by a fixed amount even when the page is fast. Use expect().to_be_visible() instead — it returns as soon as the element appears and only waits the full timeout if it never does.

Soft assertions — collect all failures at once advanced pattern
Normal assertions stop the test on the first failure. Soft assertions collect all failures and report them all at the end. Use them when you want to check multiple things on a page and see all problems at once instead of fixing one at a time.
Add to Day19_assertions.py
from playwright.sync_api import Page, expect def test_soft_assertions(page: Page): page.goto(f"{BASE}/login") # Create a soft assertion context soft = expect.soft(page) # All these run — even if one fails soft.to_have_url(f"{BASE}/login") soft.to_have_title("The Internet") # Soft assert on elements expect.soft(page.get_by_label("Username")).to_be_visible() expect.soft(page.get_by_label("Password")).to_be_visible() expect.soft(page.get_by_role("button", name="Login")).to_be_visible() # At the end — raises ALL failures at once if any failed page.context.pages[0] # soft failures reported when test ends
When to use soft vs hard assertions
# Hard assertion (default) — use for critical checks # If login URL is wrong, rest of test makes no sense → stop immediately expect(page).to_have_url(f"{BASE}/secure") # Soft assertion — use for page content checks # Check all form fields exist — want to see ALL missing fields at once expect.soft(page.get_by_label("Name")).to_be_visible() expect.soft(page.get_by_label("Email")).to_be_visible() expect.soft(page.get_by_label("Phone")).to_be_visible()
Soft assertions are especially useful for form validation testing — check that all error messages appear when you submit an empty form. Without soft assertions, you fix one error, re-run, find the next error. With soft assertions, you see all missing fields in one run.
