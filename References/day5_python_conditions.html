Conditions — making your tests smart, not just fast
A condition is a question Python answers with True or False. Every test assertion you write is a condition — "is the status code 200?", "does the error message contain 'invalid'?", "is the user active?".

You have already used conditions throughout Days 1–4. Today you go deeper — compound conditions, truthiness, guard clauses, and the ternary expression. These are the patterns that appear in every real automation codebase.

and / or / not — combining conditions
Real assertions check multiple things at once. and means both must be true. or means at least one must be true. not flips the result. These three words are how you write compound assertions — the kind that appear in every professional test suite.
Type this
# AND — both conditions must be True url = "https://staging.myapp.com" status = 200 is_valid = url.startswith("https") and status == 200 print(f"URL is secure AND status OK: {is_valid}") # True # OR — at least one must be True browser = "firefox" is_supported = browser == "chromium" or browser == "firefox" or browser == "webkit" print(f"Browser supported: {is_supported}") # True # Cleaner OR — use 'in' with a list instead is_supported = browser in ["chromium", "firefox", "webkit"] print(f"Browser supported (cleaner): {is_supported}") # True # NOT — flips the result is_logged_in = False print(f"Needs login: {not is_logged_in}") # True # Combining all three — real validation logic email = "user@test.com" password_len = 10 role = "admin" can_access_admin = ( "@" in email and password_len >= 8 and role == "admin" and not is_logged_in ) print(f"Can access admin: {can_access_admin}") # True
Output
URL is secure AND status OK: True
Browser supported: True
Browser supported (cleaner): True
Needs login: True
Can access admin: True
When you have multiple or checks against the same variable — x == "a" or x == "b" or x == "c" — always replace with x in ["a","b","c"]. Shorter, more readable, and easier to extend.

Truthiness — what Python treats as True or False trips up everyone once
In Python, you don't need to write if len(list) > 0 or if string != "". Python evaluates most values directly as True or False without an explicit comparison. Understanding this is what separates beginner Python from professional Python.
Truthy — if treats as True
True — the boolean
"hello" — any non-empty string
42 — any non-zero number
["a","b"] — any non-empty list
{"key":"val"} — any non-empty dict
object — any object that exists
Falsy — if treats as False
False — the boolean
None — nothing / not found
0 — zero
"" — empty string
[] — empty list
{} — empty dict
Type this — feel the difference
# Verbose way — beginners write this errors = [] if len(errors) > 0: print("Has errors") else: print("No errors") # prints this # Pythonic way — professionals write this if errors: print("Has errors") else: print("No errors") # same result # All the patterns you'll use in QA test_name = "test_login" result = None failures = [] passed = ["test_a", "test_b"] if test_name: # True — non-empty string print(f"Running: {test_name}") if result is None: # True — use 'is None' for None checks print("No result yet") if not failures: # True — empty list is falsy print("All tests passed!") if passed: # True — non-empty list is truthy print(f"{len(passed)} tests passed")
Output
No errors
No errors
Running: test_login
No result yet
All tests passed!
2 tests passed
Use is None / is not None explicitly for None checks — never if result when you specifically want to check for None. Why? Because 0 and "" are also falsy — if result would treat a status code of 0 as "no result". Be explicit about None.

if / elif / else — branching logic
You know the basics — today focus on the right number of branches, guard clauses, and nested conditions. These are the patterns that keep test code readable as complexity grows.
Pattern 1 — status code classifier
# Classify any HTTP status code def classify_status(code): if 200 <= code < 300: return "SUCCESS" elif 300 <= code < 400: return "REDIRECT" elif 400 <= code < 500: return "CLIENT ERROR" elif 500 <= code < 600: return "SERVER ERROR" else: return "UNKNOWN" # Test it for code in [200, 201, 301, 400, 404, 500, 503]: print(f" {code} → {classify_status(code)}")
Output
200 → SUCCESS
201 → SUCCESS
301 → REDIRECT
400 → CLIENT ERROR
404 → CLIENT ERROR
500 → SERVER ERROR
503 → SERVER ERROR
200 <= code < 300 is a Python range check — it reads exactly like maths. "code is between 200 (inclusive) and 300 (exclusive)". Much cleaner than code >= 200 and code < 300.
Pattern 2 — guard clauses (exit early)
# WITHOUT guard clauses — deeply nested, hard to read def validate_user_bad(user): if user is not None: if user.get("username"): if user.get("email") and "@" in user["email"]: return "VALID" # buried deep! # WITH guard clauses — exit early, flat and readable def validate_user(user): if user is None: return "FAIL: user is None" if not user.get("username"): return "FAIL: missing username" if not user.get("email") or "@" not in user["email"]: return "FAIL: invalid email" return "VALID" # reached only if all checks pass print(validate_user(None)) print(validate_user({"email": "test@x.com"})) print(validate_user({"username": "tom", "email": "notvalid"})) print(validate_user({"username": "tom", "email": "tom@test.com"}))
Output
FAIL: user is None
FAIL: missing username
FAIL: invalid email
VALID
Guard clauses — check the failure cases first, return early, keep the happy path last. This avoids deep nesting and makes each failure reason explicit. In test helper functions you will write dozens of these.

Ternary expression — one-line if/else use in every report
A ternary expression returns one of two values based on a condition — all in one line. You have already seen it in the challenge solutions. Today you master it properly because you will use it in every test result label, every status string, and every formatted output line.
The pattern
# Full if/else — 4 lines if condition: result = "value_if_true" else: result = "value_if_false" # Ternary — 1 line, same result result = "value_if_true" if condition else "value_if_false" # Read: "give me X if condition, otherwise Y"
Type these QA examples
# Test result labelling status = 200 label = "PASS" if status == 200 else "FAIL" print(f"Status {status}: {label}") # Environment display env = "staging" is_prod = env == "prod" safety = "⚠ PRODUCTION" if is_prod else "safe" print(f"Environment: {env} ({safety})") # Response time severity response_ms = 1850 speed = "SLOW" if response_ms > 2000 else "ACCEPTABLE" if response_ms > 1000 else "FAST" print(f"Response {response_ms}ms: {speed}") # Active/inactive display users = [ {"name": "George", "active": True}, {"name": "Emma", "active": False}, {"name": "Janet", "active": True}, ] for u in users: icon = "✓" if u["active"] else "✗" print(f" {icon} {u['name']}")
Output
Status 200: PASS
Environment: staging (safe)
Response 1850ms: ACCEPTABLE
✓ George
✗ Emma
✓ Janet
Chained ternary — "SLOW" if x > 2000 else "ACCEPTABLE" if x > 1000 else "FAST" — is fine for 3 options. More than 3 options: use if/elif/else instead. Chaining 4+ ternaries becomes unreadable fast.

QA Example 1 — Login test decision engine all patterns combined
This combines every condition concept — comparisons, and/or/not, truthiness, guard clauses, and ternary. It models the decision logic a Playwright test makes after a login attempt.
Create day5_ex1.py
def evaluate_login_result(url, title, flash, response_ms): """Evaluate a login attempt result — returns PASS or FAIL with reason.""" # Guard clauses — check failures first if not url: return "FAIL", "URL is empty" if not url.startswith("https"): return "FAIL", f"URL not secure: {url}" if not title.strip(): return "FAIL", "Page title is empty" if not flash or "secure area" not in flash.strip().lower(): return "FAIL", f"Flash message wrong: {repr(flash)}" if response_ms > 3000: return "FAIL", f"Too slow: {response_ms}ms" # All checks passed speed = "fast" if response_ms < 1000 else "acceptable" return "PASS", f"Login OK in {response_ms}ms ({speed})" # Run it against multiple scenarios scenarios = [ ("https://app.com/secure", "Secure Area", "You logged into a secure area!", 840), ("https://app.com/secure", "Secure Area", "You logged into a secure area!", 3500), ("http://app.com/secure", "Secure Area", "You logged into a secure area!", 500), ("https://app.com/secure", "Secure Area", "Your username is invalid!", 300), ("https://app.com/secure", "Secure Area", "You logged into a secure area!", 1200), ] print("Login test results:") for i, (url, title, flash, ms) in enumerate(scenarios, 1): verdict, reason = evaluate_login_result(url, title, flash, ms) icon = "✓" if verdict == "PASS" else "✗" print(f" {icon} Scenario {i}: {verdict} — {reason}")
Output
Login test results:
✓ Scenario 1: PASS — Login OK in 840ms (fast)
✗ Scenario 2: FAIL — Too slow: 3500ms
✗ Scenario 3: FAIL — URL not secure: http://app.com/secure
✗ Scenario 4: FAIL — Flash message wrong: 'Your username is invalid!'
✓ Scenario 5: PASS — Login OK in 1200ms (acceptable)
Notice the function returns two values — verdict, reason. Python lets functions return multiple values as a tuple. The caller unpacks them: verdict, reason = evaluate_login_result(...). This pattern is very clean for test helper functions that need to return both a result and an explanation.

QA Example 2 — API response classifier week 4 preview
When testing APIs you need to classify responses — not just "200 or not" but what kind of success, what kind of error, and what to do about it. This is the logic that drives smart test reporting.
Create day5_ex2.py
def assess_api_response(status, body, response_ms): """Assess an API response — classify and return assessment dict.""" # Classify the status code if 200 <= status < 300: category = "success" elif status == 401: category = "auth_error" elif status == 404: category = "not_found" elif 400 <= status < 500: category = "client_error" elif 500 <= status < 600: category = "server_error" else: category = "unknown" # Assess body has_data = bool(body) and body is not None has_error = isinstance(body, dict) and "error" in body body_ok = has_data and not has_error # Performance classification perf = ("fast" if response_ms < 500 else "ok" if response_ms < 1500 else "slow" if response_ms < 3000 else "timeout") overall = "PASS" if category == "success" and body_ok else "FAIL" return { "status": status, "category": category, "body_ok": body_ok, "perf": perf, "overall": overall } # Test it responses = [ (200, {"id": 1, "email": "a@b.com"}, 320), (404, {"error": "not found"}, 110), (200, {}, 280), (500, {"error": "server crashed"}, 4500), ] for status, body, ms in responses: r = assess_api_response(status, body, ms) print(f" {r['overall']:4} | {r['status']} {r['category']:14} | body:{r['body_ok']} | {r['perf']}")
Output
PASS | 200 success | body:True | fast
FAIL | 404 not_found | body:False | fast
FAIL | 200 success | body:False | fast
FAIL | 500 server_error | body:False | timeout
Row 3 is the most important: status 200 but body:False — the API said "OK" but returned an empty body. A test that only checks status codes would miss this bug. Conditions checking the response body are what catch silent failures that status-only checks miss.

QA Example 3 — Form field validator UI testing logic
Before a form submit in Playwright, you validate that test data is sensible. This is the exact logic that sits in a test helper function in Week 3 — validate inputs before filling the form, fail fast with a clear message if something is wrong.
Create day5_ex3.py
def validate_form_data(username, password, email, role): """Validate test form data before filling into browser.""" errors = [] # Username checks if not username or not username.strip(): errors.append("username is empty") elif len(username) < 3: errors.append(f"username too short: {len(username)} chars (min 3)") elif " " in username: errors.append("username must not contain spaces") # Password checks if not password: errors.append("password is empty") elif len(password) < 8: errors.append(f"password too short: {len(password)} chars (min 8)") # Email checks if not email: errors.append("email is empty") elif "@" not in email or "." not in email.split("@")[-1]: errors.append(f"email format invalid: {email}") # Role check if role not in ["admin", "editor", "viewer"]: errors.append(f"invalid role: {role!r}") return errors # empty = valid # Test cases test_inputs = [ ("tomsmith", "SuperSecretPassword!", "tom@test.com", "admin"), ("ab", "short", "notanemail", "superuser"), ("tom smith","ValidPass1!", "tom@test.com", "viewer"), ("", "", "", "admin"), ] for i, (u, p, e, r) in enumerate(test_inputs, 1): errs = validate_form_data(u, p, e, r) status = "VALID" if not errs else f"INVALID ({len(errs)} errors)" print(f"Input {i}: {status}") for err in errs: print(f" ✗ {err}")
Output
Input 1: VALID
Input 2: INVALID (3 errors)
✗ username too short: 2 chars (min 3)
✗ password too short: 5 chars (min 8)
✗ invalid role: 'superuser'
Input 3: INVALID (1 errors)
✗ username must not contain spaces
Input 4: INVALID (3 errors)
✗ username is empty
✗ password is empty
✗ email is empty
Notice the if / elif pattern in username checks — once one condition fails, the others are skipped. You don't report "username too short" AND "username has spaces" for the same field — the most specific failing check wins. elif is the correct tool here, not multiple if statements.

New file: day5_challenge.py. This challenge combines everything from Days 1–5 — dicts, lists, loops, strings, and conditions — into a complete test session auditor.
Build a test session auditor
Copy only this starting data:

session = {
  "env": "staging",
  "browser": "chromium",
  "base_url": "https://staging.myapp.com",
  "total_tests": 12,
  "response_times_ms": [320, 840, 1200, 450, 3500, 290, 780, 4100, 520, 670, 390, 1850],
  "results": ["PASS","PASS","FAIL","PASS","FAIL","PASS","PASS","FAIL","PASS","PASS","PASS","PASS"],
  "errors": ["Timeout on checkout", "Button not found", "API 500 on payment"]
}

Part 1 — Basic counts
Count passed, failed, pass rate %. Use .count() and ternary for overall status.

Part 2 — Response time analysis
From response_times_ms: find fastest, slowest, average (use sum() and len()). Count how many are over 2000ms (slow). Use conditions to classify the overall performance: "FAST" if average under 800, "ACCEPTABLE" if under 1500, else "SLOW".

Part 3 — Error audit
Loop over errors. For each error: check if it contains "Timeout" — label as "TIMEOUT". Check if it contains "500" — label as "SERVER ERROR". Otherwise label as "TEST FAILURE". Print each with its label.

Part 4 — Environment safety check
Check: URL starts with "https", env is not "prod", browser is in supported list ["chromium","firefox","webkit"]. Use guard clauses — return a warning message for each failure. Print all warnings or "Environment OK" if none.

Part 5 — Full audit report
Print a formatted report using everything computed above. Overall status: PASS only if pass_rate >= 80 AND no slow responses AND env is safe.
This is the most complete challenge so far. Parts 3 and 4 require you to choose between if/elif chains and guard clauses — pick the right one for each situation. When you finish, compare your choice with the ideal solution.
