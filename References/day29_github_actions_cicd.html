The workflow file — how GitHub Actions works
Everything GitHub Actions does is defined in a YAML file inside your repo. YAML uses indentation instead of brackets — spaces matter, not tabs. The file lives in a specific folder GitHub looks for automatically.
Where the file lives
playwright-ui-tests/ └── .github/ └── workflows/ └── ci.yml ← GitHub reads this automatically
YAML structure — every concept you need
name: CI Pipeline # displayed on GitHub on: # WHEN to run push: branches: [main] # on push to main branch pull_request: branches: [main] # on any PR to main jobs: # WHAT to run test: # job name (your choice) runs-on: ubuntu-latest # VM type steps: # list of steps - name: Checkout code uses: actions/checkout@v4 - name: Set up Python uses: actions/setup-python@v5 with: python-version: "3.10" - name: Install dependencies run: pip install pytest pytest-playwright requests - name: Install Playwright browsers run: playwright install --with-deps chromium - name: Run tests run: pytest --tb=short
Indentation is everything in YAML. Use 2 spaces per level — never tabs. Each - under steps is one step. uses runs a pre-built action. run runs a shell command. If indentation is wrong, GitHub will show a red error before even running your tests.

Secrets — keeping your API key safe in CI
Your reqres.in API key must never appear in your YAML file or any committed code. GitHub Secrets store sensitive values securely — they are encrypted, never shown in logs, and injected as environment variables when your workflow runs.
Step 1 — Add the secret to GitHub
1. Go to your playwright-ui-tests repo on GitHub
2. Click Settings tab
3. Click Secrets and variables → Actions
4. Click New repository secret
5. Name: REQRES_API_KEY
6. Value: paste your full API key
7. Click Add secret
Step 2 — Use the secret in your workflow
- name: Run tests run: pytest --tb=short env: REQRES_API_KEY: ${{ secrets.REQRES_API_KEY }}
Step 3 — Read it in your Python code
import os # Reads from environment — works locally AND in CI API_KEY = os.environ.get("REQRES_API_KEY", "your-local-fallback-key") HEADERS = {"x-api-key": API_KEY}
Never hardcode your API key directly in the YAML file. Never commit it to your repo. GitHub scans for exposed secrets and will warn you — but more importantly, anyone who forks or views your repo could use your key. Always use secrets.

The complete workflow file — copy this exactly
This is the production-ready workflow for your playwright-ui-tests repo. It runs API tests and UI tests separately, uploads screenshots on failure, and runs on every push.
Create this file: .github/workflows/ci.yml
name: Playwright + API Tests on: push: branches: [main] pull_request: branches: [main] workflow_dispatch: # lets you trigger manually from GitHub UI jobs: api-tests: name: API Tests runs-on: ubuntu-latest steps: - name: Checkout code uses: actions/checkout@v4 - name: Set up Python 3.10 uses: actions/setup-python@v5 with: python-version: "3.10" - name: Install dependencies run: pip install requests pytest - name: Run API tests run: pytest tests/api/ -v --tb=short env: REQRES_API_KEY: ${{ secrets.REQRES_API_KEY }} ui-tests: name: Playwright UI Tests runs-on: ubuntu-latest steps: - name: Checkout code uses: actions/checkout@v4 - name: Set up Python 3.10 uses: actions/setup-python@v5 with: python-version: "3.10" - name: Install dependencies run: pip install pytest pytest-playwright - name: Install Playwright browsers run: playwright install --with-deps chromium - name: Run UI tests run: pytest tests/ui/ -v --tb=short - name: Upload screenshots on failure if: failure() uses: actions/upload-artifact@v4 with: name: failure-screenshots path: screenshots/
workflow_dispatch adds a "Run workflow" button to your GitHub Actions tab — so you can trigger tests manually any time without pushing code. if: failure() on the screenshot upload means screenshots are only uploaded when tests fail — not on every run.

Reading CI results — passing and failing
After the workflow runs you will see either green ticks or red crosses. Here is exactly what each output means and how to debug failures.
When everything passes
GitHub Actions — success
✓ API Tests (12s)
✓ Playwright UI Tests (45s)
 
All jobs passed
When a test fails
GitHub Actions — failure
✗ Playwright UI Tests
 
FAILED tests/ui/test_login_pom.py::test_valid_login
AssertionError: Locator expected to be visible
Actual value: None
How to debug a CI failure
# 1. Click the failed job in GitHub Actions # 2. Expand the failed step — see the full pytest output # 3. Download the screenshots artifact if UI test failed # 4. Reproduce locally: pytest tests/ui/test_login_pom.py::test_valid_login -v --headed
--tb=short in the workflow gives you just the key failure lines — not the full traceback. For more detail during debugging change it to --tb=long. For a quick summary in CI output, --tb=short is correct.

requirements.txt — the professional way to manage dependencies
Right now your workflow installs packages by listing them in the YAML. The professional way is a requirements.txt file — one file lists all dependencies, CI installs from it, and anyone cloning your repo can run the same setup.
Create requirements.txt in your repo root
pytest pytest-playwright requests python-dotenv
Update your workflow to use it
- name: Install dependencies run: pip install -r requirements.txt
Update your README.md run instructions
## Setup ```bash pip install -r requirements.txt playwright install chromium pytest -v ```
Anyone who clones your repo now runs pip install -r requirements.txt and gets exactly the same packages. No manual figuring out what to install. This is the standard for every Python project.
Use python-dotenv for local secrets
# Create .env file (add to .gitignore — never commit this) # .env REQRES_API_KEY=your-actual-key-here
import os from dotenv import load_dotenv load_dotenv() # loads .env file locally — ignored in CI (uses GitHub Secret) API_KEY = os.environ.get("REQRES_API_KEY", "") HEADERS = {"x-api-key": API_KEY}
Make sure .env is in your .gitignore. Run echo ".env" >> .gitignore to add it. If you accidentally commit your API key, rotate it immediately at app.reqres.in.
